# 04-工具链组合工作流

> 状态：已完成
> 适用版本：ocorm 3.x（当前仓库 3.0.2）
> 最后更新：2026-03-27

## 1. 目标
这一页只讨论三个工具的协作边界：
- `MigrationGenerator`：把 `SchemaDiffer` 产出的差异转成 `GeneratedMigration`。
- `defineFactory` / `EntityFactory`：构造符合元数据约束的测试或种子数据。
- `SeederManager`：顺序执行 Seeder，并汇总每个 Seeder 的成功/失败结果。

推荐把它们组合成一个明确的流水线，而不是混成一个脚本：
1. 先生成并审阅迁移文本。
2. 再验证数据形状是否满足最新实体元数据。
3. 最后由 `SeederManager.run()` 执行具体 Seeder。

## 2. 工具职责边界
从源码行为看，这三个工具没有互相替你兜底：
- `MigrationGenerator` 不落盘、不执行 SQL，只返回 `GeneratedMigration`。
- `EntityFactory.make/makeMany` 只构造 `EntityData`；`create/createMany` 才会走 `Repository.save()`。
- `SeederManager.run()` 逐个执行 Seeder；单个 Seeder 失败不会中断后续 Seeder。

这意味着组合工作流要自己控制时机：
- 需要“预览数据”时，用 `make/makeMany`。
- 需要“真正入库”时，先保证数据库已初始化，再用 `create/createMany` 或在 Seeder 内手动 `repo.save(...)`。
- 需要“稳定测试”时，给 `MigrationGenerator.generateWithStore()` 注入自定义 `schemaDiffer`，不要把真实库状态直接耦合进单测。

## 3. 推荐流程
最稳妥的顺序是：
- `SchemaDiffer` 或自定义 `SchemaDiffResult` 先产出差异。
- `MigrationGenerator` 生成迁移文本并检查 `hasChanges`、`upSql`、`downSql`。
- `defineFactory` 基于当前实体元数据生成数据草稿。
- `SeederManager` 统一调度 Seeder，把结果收口到 `SeederExecutionResult`。

这个顺序的原因很简单：
- 如果 Schema 还没确认，Factory 生成出来的数据字段也没有意义。
- 如果 Seeder 先跑，而表结构还没定型，失败点会分散到 Repository/数据库层，排查成本更高。
- `SeederManager` 没有事务级全局回滚语义，不能拿它当“迁移 + 填数”的原子编排器。

## 4. 最小可运行示例
下面的例子只使用仓库里现有 API，并刻意采用“预览模式”：
- `MigrationGenerator` 使用自定义 `SchemaDiffer`，避免依赖真实 `RdbStore`
- `EntityFactory` 使用 `makeMany()`，避免触发数据库写入
- `SeederManager` 负责顺序执行一个 Seeder，并返回结果对象

```ets
import { relationalStore } from '@kit.ArkData'
import { defineEntity } from '../../src/main/ets/core/EntitySchema'
import { ColumnType } from '../../src/main/ets/types/ColumnType'
import { Repository } from '../../src/main/ets/repository/Repository'
import { defineFactory } from '../../src/main/ets/tools/Factory'
import { Seeder, SeederManager } from '../../src/main/ets/tools/SeederManager'
import { MigrationGenerator } from '../../src/main/ets/tools/MigrationGenerator'
import { SchemaDiffResult, SchemaDiffer, SchemaChangeType } from '../../src/main/ets/schema/SchemaDiffer'

defineEntity('User', {
  columns: [
    { property: 'id', type: ColumnType.INTEGER, primaryKey: true },
    { property: 'name', type: ColumnType.TEXT },
    { property: 'email', type: ColumnType.TEXT }
  ]
})

const userFactory = defineFactory('User', (faker, index) => ({
  id: index + 1,
  name: faker.name(),
  email: faker.email('example.com')
}))

class MockSchemaDiffer extends SchemaDiffer {
  private result: SchemaDiffResult

  constructor(result: SchemaDiffResult) {
    super()
    this.result = result
  }

  async diff(_store: relationalStore.RdbStore): Promise<SchemaDiffResult> {
    return this.result
  }
}

class PreviewUserSeeder implements Seeder {
  async run(_repo: Repository): Promise<void> {
    const drafts = await userFactory.makeMany(2, (index) => ({
      name: `preview-${index}`
    }))
    console.info(`draftCount=${drafts.length}`)
  }
}

const diff = new SchemaDiffResult()
diff.addChange({
  type: SchemaChangeType.CREATE_TABLE,
  tableName: 'users',
  sql: ['CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT, email TEXT)'],
  rollbackSql: ['DROP TABLE users'],
  summary: 'create users',
  destructive: false
})

const generated = await MigrationGenerator.generateWithStore(
  {} as relationalStore.RdbStore,
  {
    schemaDiffer: new MockSchemaDiffer(diff),
    classNamePrefix: 'WorkflowMigration',
    fileExtension: 'ets'
  }
)

const result = await SeederManager.run([new PreviewUserSeeder()])

console.info(generated.fileName)
console.info(generated.hasChanges)
console.info(result.success)
console.info(result.items[0].seederName)
```

这个示例能证明三件事：
- 迁移文本可以先生成再审阅。
- Factory 可以先校验数据形状，不必立刻写库。
- SeederManager 的价值在于调度和结果汇总，而不是替你管理 Schema 生命周期。

## 5. 真实入库时的切换点
从预览模式切到真实入库时，只改两处：
- Factory 从 `make/makeMany` 切到 `create/createMany`
- 或者在 Seeder 的 `run(repo)` 里显式调用 `repo.save(...)`

但前提不变：
- 实体元数据必须已注册。
- `DatabaseManager.getInstance().getStore()` 必须能拿到真实连接。
- 目标表结构必须已经由你的迁移流程或建表流程准备好。

## 6. 常见误用
下面三个误用是最常见的，也是组合工作流里最容易浪费时间的地方。

```ets
import { relationalStore } from '@kit.ArkData'
import { defineFactory } from '../../src/main/ets/tools/Factory'
import { Seeder, SeederManager } from '../../src/main/ets/tools/SeederManager'
import { Repository } from '../../src/main/ets/repository/Repository'
import { MigrationGenerator } from '../../src/main/ets/tools/MigrationGenerator'

// 误用 1：没有初始化数据库就直接 createMany()
const factory = defineFactory('User', () => ({
  id: 1,
  name: 'alice'
}))
await factory.createMany(2)

// 误用 2：给 generateWithStore() 传假 store，却不注入 schemaDiffer
await MigrationGenerator.generateWithStore({} as relationalStore.RdbStore)

// 误用 3：对象字面量 seeder 不显式给 entityName，推断结果会落到 Object
const badSeeder: Seeder = {
  async run(_repo: Repository): Promise<void> {
    return
  }
}
await SeederManager.run([badSeeder])
```

对应后果：
- 误用 1：`create/createMany` 会走 `Repository.save()`，最终触发数据库连接或表结构问题。
- 误用 2：默认 `SchemaDiffer` 会访问真实 `store.querySql(...)`，空对象会在运行时失败。
- 误用 3：`SeederManager` 会把类名推断成 `Object`，最终构造 `new Repository('Object')`，大概率找不到实体元数据。

## 8. 验收清单
- `MigrationGenerator` 的输出只被当成文本快照使用，不被误认为自动执行器。
- `EntityFactory.make/makeMany` 只用于预览和装配数据形状。
- 真正落库时，已经提前准备好数据库连接和表结构。
- `SeederManager.run()` 的返回值被检查，而不是只看控制台输出。
- CI 使用自定义 `SchemaDiffer` 或固定 `SchemaDiffResult`，避免测试依赖真实数据库状态。
