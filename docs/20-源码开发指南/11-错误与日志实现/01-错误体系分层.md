# 01-错误体系分层

> 状态：已完成
> 适用版本：ocorm 3.x（当前 3.0.2）
> 最后更新：2026-03-27

## 1. 目标
- 统一 OCORM 错误分层认知：`OrmError` 基类 -> 领域子类 -> 场景化具体错误。
- 明确错误对象最小约定：`code`、`context`、本地化后的 `message`。
- 给出可落地的处理、误处理、观测排障方法，避免错误吞没和敏感信息泄露。

## 2. 分层模型
OCORM 当前错误体系是“继承 + 错误码 + 上下文”三件套：
- 基座层：`OrmError` + `ErrorContext`（`errors/OrmError.ets`）。
- 领域层：`DatabaseError`、`QueryError`、`TransactionError`、`ValidationError` 等（`errors/*.ets`）。
- 场景层：`ConnectionError`、`ExecutionError`、`InvalidConditionError`、`HookExecutionError` 等具体错误。

错误码按区间划分（`errors/ErrorCodes.ets`）：
- `ORM001-ORM099` 元数据
- `ORM101-ORM199` 查询
- `ORM201-ORM299` 映射
- `ORM301-ORM399` 事务
- `ORM401-ORM499` 数据库
- `ORM501-ORM599` 关联
- `ORM601-ORM699` 钩子
- `ORM701-ORM799` 软删除
- `ORM801-ORM899` 校验

```ts
import { OrmError, ErrorContext } from 'ocorm'
import { QueryError, InvalidConditionError } from 'ocorm'
import { ConnectionError } from 'ocorm'

const ctx = new ErrorContext()
ctx.entityName = 'User'
ctx.operation = '构建查询条件'

const base = new OrmError('基础错误', 'ORM999', ctx)
const query = new QueryError('查询错误', 'ORM101', ctx)
const concrete = new InvalidConditionError('age', '不支持 null 比较')
const db = new ConnectionError('ocorm.db', 'permission denied')
```

## 3. 正确处理方式
原则：
- 业务边界只捕获 `OrmError`，再统一读取 `code` 和 `context`。
- 报错时使用 `getFullMessage()` 补齐上下文，不丢失定位信息。
- 记录日志时使用 `Logger.logError()`，由日志层做敏感信息脱敏。

```ts
import { OrmError } from 'ocorm'
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.INFO)

function handleOrmFailure(e: unknown): void {
  if (e instanceof OrmError) {
    logger.logError(e.getFullMessage())
    return
  }
  logger.logError(`未知异常: ${String(e)}`)
}
```

## 4. 误处理示例（禁止）
典型反模式：
- 只记录 `error.message`，丢失 `code/context`。
- 直接拼接原始 SQL 与异常文本，不经过脱敏。
- 把所有错误一律转成字符串后再抛出，破坏错误分层。

```ts
// 反例：丢失层级与上下文，且可能泄露敏感数据
try {
  // ... query
} catch (e) {
  // 错误：没有判断 OrmError
  // 错误：没有记录 code/context
  // 错误：可能直接输出敏感值
  console.error((e as Error).message)
  throw new Error(String(e))
}
```

## 5. 观测与排障方式
推荐流程：
1. 用 `Logger.configure(true, LogLevel.DEBUG)` 临时提高日志等级复现问题。
2. 检查 `OrmError.code` 对应区间，先定层（查询/事务/数据库/校验）。
3. 读取 `context` 的 `entityName/columnName/tableName/operation/details` 精确定位。
4. 使用 `getFullMessage()` 与日志历史联动（测试场景可用 `setRecordHistory(true)` + `getLogHistory()`）。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)
logger.setRecordHistory(true)

// 复现后提取日志轨迹
const history = logger.getLogHistory()
for (let i = 0; i < history.length; i++) {
  console.info(history[i])
}
```

## 6. 与国际化和脱敏的关系
- `OrmError` 构造时调用 `formatErrorMessage(...)`，消息可根据 `setErrorLocale(...)` 切换语言（`errors/ErrorLocale.ets`）。
- `DatabaseError`、`QueryError`、`TransactionError`、`HookExecutionError` 内部会调用 `sanitizeErrorMessage` / `sanitizeSqlPreview`（`errors/ErrorSanitizer.ets`），用于敏感值与路径脱敏。

```ts
import { ExecutionError, setErrorLocale } from 'ocorm'

setErrorLocale('en')
const err = new ExecutionError("SELECT * FROM users WHERE phone='13800138000'", 'db timeout')
console.info(err.message) // 本地化 + 脱敏后的文本
```

## 7. 验收清单
- 是否统一使用 `OrmError` 分层而不是裸 `Error`。
- 是否保留并传递 `code` 与 `context`。
- 是否通过 `Logger` 记录并避免敏感信息直出。
- 是否可通过错误码区间快速回到对应源码层排障。
