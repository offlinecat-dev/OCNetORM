# 02-错误码与错误上下文

> 状态：已完成
> 适用版本：ocorm 3.x（当前 3.0.2）
> 最后更新：2026-03-27

## 1. 目标
- 说明 `errors/ErrorCodes.ets` 中错误码区间的实际含义，以及它们与具体错误类的对应关系。
- 说明 `errors/OrmError.ets` 中 `ErrorContext` 的字段职责，避免只拿 `message` 排障。
- 给出正确处理、误处理、观测排障的固定做法，确保错误码、上下文、日志三者能够串起来。

## 2. 结论先行
OCORM 当前的错误识别主键不是错误类名，而是 `OrmError.code`。`OrmError` 在构造时持有 `code` 与 `context`，`getFullMessage()` 再把 `entityName`、`columnName`、`tableName`、`operation`、`details` 拼成可排障文本，因此脱离这两个字段单独看 `message` 基本没有工程价值。

## 3. 错误码区间
`errors/ErrorCodes.ets` 把错误码按区间分组：
- `ORM001-ORM099`：元数据。
- `ORM101-ORM199`：查询。
- `ORM201-ORM299`：映射。
- `ORM301-ORM399`：事务。
- `ORM401-ORM499`：数据库。
- `ORM501-ORM599`：关联关系。
- `ORM601-ORM699`：钩子。
- `ORM701-ORM799`：软删除。
- `ORM801-ORM899`：校验。

当前源码中的典型映射如下：
- `ORM101` -> `InvalidConditionError`
- `ORM102` -> `PredicateBuildError`
- `ORM103` -> `InvalidOperatorError`
- `ORM104` -> `SubQueryError`、`ScopeNotFoundError`
- `ORM301` -> `TransactionNotStartedError`
- `ORM302` -> `TransactionRollbackError`
- `ORM303` -> `TransactionCommitError`
- `ORM401` -> `ConnectionError`
- `ORM402` -> `ExecutionError`
- `ORM403` -> `DatabaseNotInitializedError`
- `ORM501` -> `RelationNotFoundError`
- `ORM504` -> `JoinTableNotFoundError`
- `ORM601` -> `HookExecutionError`
- `ORM801-ORM814` -> `ValidationError` 及其细分子类

```ts
import {
  ERROR_INVALID_CONDITION,
  ERROR_TRANSACTION_COMMIT_FAILED,
  ERROR_SQL_EXECUTION_FAILED,
  ERROR_VALIDATION_REQUIRED
} from 'ocorm'

console.info(ERROR_INVALID_CONDITION) // ORM101
console.info(ERROR_TRANSACTION_COMMIT_FAILED) // ORM303
console.info(ERROR_SQL_EXECUTION_FAILED) // ORM402
console.info(ERROR_VALIDATION_REQUIRED) // ORM802
```

## 4. ErrorContext 字段约定
`errors/OrmError.ets` 中 `ErrorContext` 只有 5 个字段：
- `entityName`：实体名。
- `columnName`：列名。
- `tableName`：表名或数据库名。
- `operation`：执行动作，例如 `数据库连接`、`事务提交`、`构建查询条件`。
- `details`：补充说明，通常会写失败原因、脱敏后的 SQL 片段、长度范围等。

这些字段不是全部必填，但谁构造错误谁就负责填。典型填充模式：
- `ConnectionError`：填 `tableName`、`operation`、`details`。
- `InvalidConditionError`：填 `columnName`、`operation`、`details`。
- `RequiredValidationError`：填 `entityName`、`columnName`、`operation`。
- `HookExecutionError`：填 `entityName`、`operation`、`details`。

```ts
import { ErrorContext, OrmError } from 'ocorm'

const context = new ErrorContext()
context.entityName = 'User'
context.columnName = 'email'
context.operation = '数据校验'
context.details = '字段缺失'

const error = new OrmError('字段 email 为必填项', 'ORM802', context)
console.info(error.code)
console.info(error.context.columnName)
console.info(error.getFullMessage())
```

## 5. 正确处理方式
正确做法只有一条主线：捕获 `OrmError`，先用 `code` 定位层，再用 `context` 定位点，最后用 `Logger.logError()` 留痕。
- 先判断 `e instanceof OrmError`，不要假设所有失败都是裸 `Error`。
- 先看 `code` 所在区间，再看 `context.operation/details`。
- 对外返回时可以裁剪信息；对内日志必须保留 `getFullMessage()`。

```ts
import { OrmError } from 'ocorm'
import { InvalidConditionError } from 'ocorm'
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.INFO)

function runQueryGuard(): void {
  try {
    throw new InvalidConditionError('age', '不支持 null 比较')
  } catch (e) {
    if (e instanceof OrmError) {
      logger.logError(e.getFullMessage())
      if (e.code >= 'ORM101' && e.code <= 'ORM199') {
        logger.logWarn(`查询层失败，操作=${e.context.operation}`)
      }
      return
    }
    logger.logError(`未知异常: ${String(e)}`)
  }
}
```

## 6. 误处理示例
下面这些写法都在主动摧毁可观测性：
- 只打 `message`，不记录 `code/context`。
- 把 `OrmError` 包装成新的 `Error` 再抛，导致调用方丢失错误层级。
- 把 `details` 当成最终用户文案直接展示，而不是当作排障上下文。

```ts
import { ExecutionError } from 'ocorm'

try {
  throw new ExecutionError("SELECT * FROM users WHERE phone='13800138000'", 'db timeout')
} catch (e) {
  // 反例 1：只看 message，不看 code
  console.error((e as Error).message)

  // 反例 2：重新抛裸 Error，丢失上下文
  throw new Error(String((e as Error).message))
}
```

## 7. 观测与排障方式
排障顺序建议固定为：
1. 通过 `Logger.configure(true, LogLevel.DEBUG)` 打开充分日志。
2. 用 `logger.setRecordHistory(true)` 保留最近日志，避免只看控制台碎片。
3. 抓到异常后同时看 `code`、`context`、`getFullMessage()`。
4. 若 `code` 相同但来源不同，优先比较 `operation` 与 `details`。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'
import { TransactionCommitError } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)
logger.setRecordHistory(true)

const error = new TransactionCommitError('database is locked')
logger.logError(error.getFullMessage())

const history = logger.getLogHistory()
for (let i = 0; i < history.length; i++) {
  console.info(history[i])
}
```

## 8. 代码评审检查项
- 是否存在 `catch (e) { throw new Error(...) }` 这类上下文丢失代码。
- 是否基于 `code` 区间做了最小限度的错误分流。
- 是否在需要追踪的边界上记录 `getFullMessage()` 而不是裸 `message`。
- 是否把 `entityName`、`columnName`、`operation` 这类定位字段正确写入 `ErrorContext`。

## 9. 常见问题
### 9.1 为什么同一个错误码会对应多个错误类
因为错误码表达的是错误域，不一定是一对一场景标识。比如 `ORM104` 同时被 `SubQueryError` 和 `ScopeNotFoundError` 使用，区分它们要看 `name` 和 `context.details`。

### 9.2 为什么 `getFullMessage()` 比 `message` 更重要
因为 `message` 只保证最终展示文本，`getFullMessage()` 才会串上 `code` 与上下文字段。线上排障如果只留 `message`，基本等于主动降级为人工猜测。
