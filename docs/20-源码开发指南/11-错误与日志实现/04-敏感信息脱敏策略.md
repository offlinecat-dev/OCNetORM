# 04-敏感信息脱敏策略

> 状态：已完成
> 适用版本：ocorm 3.x（当前 3.0.2）
> 最后更新：2026-03-27

## 1. 目标
- 说明 `errors/ErrorSanitizer.ets` 当前真实提供的脱敏入口与规则。
- 说明错误类与日志类在哪些位置已经自动接入脱敏，哪些位置仍然要求调用方自行处理。
- 给出正确处理、误处理、观测排障方法，避免把 SQL 字面值、病历号、手机号、路径直接打到日志或错误消息里。

## 2. 源码事实
当前对外可直接使用的脱敏 API 只有两个：
- `sanitizeErrorMessage(rawMessage)`：通用错误文本脱敏，最大长度 256。
- `sanitizeSqlPreview(rawSql)`：SQL 预览脱敏，内部先走 `sanitizeErrorMessage`，再把结果截断到 120。

核心规则来自 `errors/ErrorSanitizer.ets`：
- 清理控制字符 `\x00-\x1F` 和 `\x7F`。
- 脱敏敏感键值对，例如 `patientId=...`、`mrn=...`、`phone=...`、`visit_no=...`。
- 将单引号和双引号中的值替换成 `"[***]"` 或 `'[***]'`。
- 将长度大于等于 4 的纯数字 token 替换成 `[***]`；像 `123` 这种短数字不会被替换。
- 将 Windows 路径和类 Unix 路径替换成 `[path]`。
- 压缩多余空白，并按长度做截断。

```ts
import {
  sanitizeErrorMessage,
  sanitizeSqlPreview
} from 'ocorm'

const rawMessage = 'Save failed: patientId=abc123, mrn=9988, file=E:\\data\\prod.db'
const rawSql = "SELECT * FROM visits WHERE visit_no = 1234 AND phone = '13800138000'"

console.info(sanitizeErrorMessage(rawMessage))
console.info(sanitizeSqlPreview(rawSql))
```

## 3. 自动接入点
当前源码里，以下错误类会在构造阶段自动调用脱敏逻辑：
- `ConnectionError`：对 `reason` 调用 `sanitizeErrorMessage`。
- `ExecutionError`：对 `sql` 调用 `sanitizeSqlPreview`，对 `reason` 调用 `sanitizeErrorMessage`。
- `InvalidConditionError`、`PredicateBuildError`、`SubQueryError`：对失败原因做 `sanitizeErrorMessage`。
- `TransactionRollbackError`、`TransactionCommitError`：对失败原因做 `sanitizeErrorMessage`。
- `HookExecutionError`：对原始异常文本做 `sanitizeErrorMessage`。

日志层的自动接入点在 `logging/Logger.ets`：
- `logQuery(sql, duration)`：只在 `DEBUG` 级别记录，内部会调用 `sanitizeSql(...)`，掩码字符串字面量、敏感键值对和 4 位及以上数字。
- `logError(errorMessage)`、`logWarn(warnMessage)`：会以 `redactSensitive=true` 进入 `formatMessage(...)`，额外掩码路径、引号内容、长数字和敏感键值对。
- `logInfo(...)`、`logDebug(...)`：不会自动做敏感脱敏，调用方不要把原始秘密直接塞进去。

## 4. 正确处理方式
正确做法很明确：
- 已有错误类支持自动脱敏时，直接传原始输入给错误构造器，不要自己拼一段未脱敏说明再丢进日志。
- 需要手动记录的字符串，先走 `sanitizeErrorMessage` 或 `sanitizeSqlPreview`。
- 面向日志时优先走 `Logger.logError()`、`Logger.logWarn()`、`Logger.logQuery()`，不要绕开 Logger 直打原文。

```ts
import { ExecutionError, ConnectionError } from 'ocorm'
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'
import { sanitizeErrorMessage } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)

const rawReason = 'open E:\\patient\\prod.db failed: mrn=9988, operator="alice"'
const dbError = new ConnectionError('prod.db', rawReason)
logger.logError(dbError.getFullMessage())

const sqlError = new ExecutionError(
  "SELECT * FROM visits WHERE visit_no = 1234 AND phone = '13800138000'",
  rawReason
)
logger.logError(sqlError.getFullMessage())
logger.logWarn(sanitizeErrorMessage('fallback warning: patientId=abc123, path=/data/visit/001'))
```

## 5. 误处理示例
以下写法都在主动制造泄露：
- 直接把原始 SQL 字符串送进 `console.error` 或裸 `hilog`。
- 以为 `logInfo()`、`logDebug()` 会自动脱敏，然后塞入病历号、路径、手机号。
- 自己手工拼接一段 `details`，但完全不走 `sanitizeErrorMessage`。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)

const sql = "SELECT * FROM visits WHERE visit_no = 1234 AND phone = '13800138000'"
const reason = 'save failed: patientId=abc123, file=E:\\secret\\prod.db'

// 反例 1：直接输出原始 SQL
console.error(sql)

// 反例 2：误以为 logDebug 自动脱敏
logger.logDebug(`raw debug: ${reason}`)

// 反例 3：手工拼接未脱敏详情
logger.logInfo(`manual info: ${sql} | ${reason}`)
```

## 6. 观测与排障方式
验证脱敏是否生效，不要肉眼猜。建议用 `Logger` 历史缓冲做检查：
1. `logger.configure(true, LogLevel.DEBUG)`，确保查询日志也会被记录。
2. `logger.setRecordHistory(true)`，保留最近 200 条日志。
3. 触发一次带敏感内容的 `logQuery`、`logError` 或具体错误类。
4. 从 `getLogHistory()` 中确认是否只剩 `'[***]'`、`"[***]"`、`[***]`、`[path]`。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)
logger.setRecordHistory(true)
logger.clearLogHistory()

logger.logQuery("SELECT * FROM visits WHERE visit_no = 1234 AND phone = '13800138000'", 25)
logger.logError('Save failed: patientId=abc123, mrn=9988, path=/data/visit/001')

const history = logger.getLogHistory()
for (let i = 0; i < history.length; i++) {
  console.info(history[i])
}
```

## 7. 行为边界
需要明确几条边界，避免误判：
- `sanitizeErrorMessage` 只会替换长度大于等于 4 的纯数字 token，`123` 这种短数字会保留。
- `sanitizeSqlPreview` 不是完整 SQL parser，它只负责“预览级别”的文本脱敏与截断。
- `Logger.logQuery()` 会做 SQL 相关掩码，但不处理你在 `logDebug()`/`logInfo()` 里主动塞进去的原始内容。
- 脱敏不等于最小暴露。即使已经脱敏，生产环境也不应长期打开 `DEBUG` 级别查询日志。

```ts
import {
  sanitizeErrorMessage,
  sanitizeSqlPreview
} from 'ocorm'

console.info(sanitizeErrorMessage('id = 123'))
console.info(sanitizeErrorMessage('visit_no = 1234'))
console.info(sanitizeSqlPreview("INSERT INTO users(name) VALUES('alice')"))
```

## 8. 代码评审检查项
- 是否所有手工拼接的错误详情都先经过 `sanitizeErrorMessage`。
- 是否将原始 SQL 统一交给 `ExecutionError` 或 `Logger.logQuery()` 而不是自行打印。
- 是否把需要脱敏的错误/告警统一走 `logError()`、`logWarn()` 而非 `logInfo()`、`logDebug()`。
- 是否在生产环境避免长期开启 `DEBUG` 级别 SQL 日志。

## 9. 常见问题
### 9.1 为什么日志里还能看到 `123`
因为当前规则只掩码 4 位及以上的数字 token。短数字通常用于页码、状态码或小整数，源码没有把它们全部视为敏感信息。

### 9.2 为什么 `ValidationError` 文案里可能还会出现业务原文
因为 `ValidationError` 系列本身不统一负责脱敏。需要自行构造自定义消息时，调用方必须先做 `sanitizeErrorMessage`，否则你传什么它就保留什么。
