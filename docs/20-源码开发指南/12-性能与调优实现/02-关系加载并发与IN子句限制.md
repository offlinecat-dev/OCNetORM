# 02-关系加载并发与IN子句限制

> 状态：已完成
> 适用版本：ocorm 3.x（当前仓库实现）
> 最后更新：2026-03-28

## 1. 目标
- 说明 `RelationLoader` 当前如何做批量关系加载。
- 说明 `relationInClauseLimit` 对 SQL 分批的直接影响。
- 说明当前仓库“可调参数”和“内部实现细节”的边界，不虚构不存在的并发旋钮。

## 2. 当前仓库能调什么
关系加载这块，当前仓库公开、稳定的调优面主要是：
- `DatabaseConfig.setRelationInClauseLimit(limit)`
- `DatabaseConfig.setRelationLoadConcurrency(limit)`
- `RelationLoader.setInClauseLimit(limit)`（底层级别旋钮）
- 查询/关系加载自身的 timeout 行为

```arkts
import { DatabaseConfig } from 'ocorm'

const config = new DatabaseConfig('app.db')
  .setRelationInClauseLimit(200)
  .setRelationLoadConcurrency(4)
```

```arkts
import { RelationLoader } from 'ocorm'

RelationLoader.setInClauseLimit(300)
```

并发加载虽然由 `RelationLoaderParallelSupport` 执行，但并发上限并不是“完全不可配”；当前公开配置入口就是 `DatabaseConfig.setRelationLoadConcurrency(n)`。

## 3. 正确示例
当 one-to-one / many-to-many 关系批量加载规模很大时，应该先控制 `IN (...)` 批次大小，再观察 SQL 次数与总时长。

```arkts
import { DatabaseConfig, RelationLoader } from 'ocorm'

const batchSize = 100
const config = new DatabaseConfig('one_to_one_batch.db')
  .setRelationInClauseLimit(batchSize)
  .setRelationLoadConcurrency(4)
const loader = new RelationLoader('User')

console.info(config.relationInClauseLimit, loader)
```

如果你要走并行关系加载，当前入口仍然是 `loadRelationsParallel(...)`。

```arkts
import { RelationLoader } from 'ocorm'

const loader = new RelationLoader('User')
const entities = await repo.findAll()

await loader.loadRelationsParallel(entities, ['profile', 'posts'])
```

从 `EntryOrmPerformanceBatchSuite` 的测试口径看，批量关系加载应按 `relationInClauseLimit` 自动分批，而不是把所有主键一次性塞进一个巨型 `IN (...)`。

```text
大批量加载验收点:
1. SQL 次数随 relationInClauseLimit 分批增长
2. 单条 SQL 的 IN 列表不会无限膨胀
3. 关系结果不因分批而丢失
```

## 4. 误用示例
把 `relationInClauseLimit` 设得过大，不会让加载“无限快”，只会把单条 SQL 做大，可能放大解析和执行成本。

```arkts
const config = new DatabaseConfig('app.db').setRelationInClauseLimit(100000)
```

把它设得过小也不对，会把一批关系拆成过多 SQL。

```arkts
const config = new DatabaseConfig('app.db').setRelationInClauseLimit(1)
```

另一个误用是把“内部支持并行加载”理解成“外部可以随便并发 new 多个 loader 同时跑同一批数据”。如果底层连接池、查询槽位和 timeout 没跟上，只会把查询行为打散得更难排查。

## 5. 观测与复现
建议直接用当前仓库已有用例作为复现实验入口：
- `EntryOrmPerformanceBatchSuite`：验证分批行为
- `EntryRelationTimeoutSuite`：验证关系加载超时边界

```bash
rg -n "relationInClauseLimit|RelationLoader|loadRelationsParallel" OCORM/src/main/ets/query entry/src/main/ets/suites
```

## 6. 调优建议
- 默认先从 `100 ~ 500` 范围试 `relationInClauseLimit`
- 关系维度多时优先看 SQL 次数和总耗时，不要只盯单次响应
- 一旦出现“关系部分丢失”“超时跳过加载”，先排查 timeout 与分批，不要先怀疑映射层
