# 03-缓存参数调优

> 状态：已完成
> 适用范围：`QueryCache` 与 `DatabaseConfig.setQueryCache`
> 最后更新：2026-03-28

## 1. 目标
- 说明当前仓库缓存真正能调的参数。
- 给出 `maxSize`、`ttlMs`、开关行为的调优方法。
- 明确缓存命中不是越大越好，失效策略同样重要。

## 2. 当前可调参数
公开调优面来自两层：
- `DatabaseConfig.setQueryCache(enabled, maxSize?, ttlMs?)`
- `QueryCache.configure({ enabled, maxSize, ttlMs })`

```arkts
import { DatabaseConfig } from 'ocorm'

const config = new DatabaseConfig('app.db')
  .setQueryCache(true, 256, 60000)
```

```arkts
import { QueryCache } from 'ocorm'

const cache = QueryCache.getInstance()
cache.configure({ enabled: true, maxSize: 100, ttlMs: 60000 })
cache.clear()
```

## 3. 正确示例
调优时先保证行为正确，再谈命中率。

```arkts
const cache = QueryCache.getInstance()
cache.configure({ enabled: true, maxSize: 200, ttlMs: 30000 })

const cached = cache.get('User', 1)
console.info(cached)
```

实体级失效应优先使用精准失效，而不是动不动全清。

```arkts
cache.invalidate('User', 1)
cache.invalidateEntity('User')
```

从 `CacheSuite` 的测试口径看，当前缓存需要重点关注：
- TTL 过期行为
- `invalidateEntity(...)`
- `clear()`
- 统计计数与命中率

```text
最小观测项:
1. enabled 是否生效
2. maxSize 是否导致逐出
3. ttlMs 是否导致准时过期
4. invalidate / clear 是否与写入路径一致
```

## 4. 误用示例
TTL 设得太长，可能把过期业务数据留太久。

```arkts
cache.configure({ enabled: true, maxSize: 1000, ttlMs: 24 * 60 * 60 * 1000 })
```

TTL 设得太短，又会把缓存退化成几乎不命中。

```arkts
cache.configure({ enabled: true, maxSize: 100, ttlMs: 1 })
```

还有一种误用是高频写入路径都走 `rawExecute`，因为 `rawExecute` 成功后会 `queryCache.clear()`，缓存参数再好也会被全部打掉。

## 5. 复现与验证
直接复用当前仓库 `CacheSuite` 的能力验证：

```bash
rg -n "QueryCache_|configure\\(|invalidate\\(|invalidateEntity\\(|clear\\(" ../entry/src/main/ets/suites/CacheSuite.ets
```

## 6. 调优建议
- 初始值可从 `maxSize=100~300`、`ttlMs=30000~60000` 起步
- 读多写少场景才值得把 `maxSize` 拉大
- 如果命中率差而 `clear()` 次数很多，先排查写路径而不是盲目调大缓存
