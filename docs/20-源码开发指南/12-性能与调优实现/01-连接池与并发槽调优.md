# 01-连接池与并发槽调优

> 适用范围：当前 `OCORM` 连接池、查询槽位门控与连接租约实现
>
> 核心源码：`DatabaseConfig.ets`、`DatabaseManager.ets`、`FixedStorePool.ets`、`StoreLease.ets`、`QuerySlotGate.ets`、`StorePoolFactory.ets`、`QueryExecutor.ets`、`EntryConnectionPoolSuite.ets`

## 1. 先说结论

- 连接池真正影响吞吐的主参数只有 4 个：`enabled`、`minSize`、`maxSize`、`acquireTimeoutMs`
- 查询并发门控真正影响排队的主参数只有 2 个：`maxConcurrentQueries`、`queryTimeoutMs`
- `connectionPool.idleTimeoutMs` 与 `connectionPool.healthCheckIntervalMs` 当前会被保存和比较，但 `FixedStorePool` 没有实现空闲回收和池内健康探测，不要把它们当成已经生效的调优旋钮

## 2. 实现事实

### 2.1 连接池怎么工作

- `StorePoolFactory.createPool()` 在 `connectionPool.enabled=true` 时创建 `FixedStorePool`，随后立即执行 `warmUp()`
- `warmUp()` 会把池子补到 `minSize`
- `acquire()` 优先拿空闲连接，其次在 `currentSize < maxSize` 时新建连接，否则进入等待队列
- 等待队列超时后抛出 `获取连接超时`
- `metrics()` 可观测 `currentSize`、`activeLeases`、`idleStores`、`pendingRequests`、`timeoutCount`

### 2.2 查询槽位怎么工作

- `QuerySlotGate.configure()` 从 `DatabaseConfig` 读取 `maxConcurrentQueries` 和 `queryTimeoutMs`
- `maxConcurrentQueries=0` 表示不限制
- 当活跃槽位达到上限时，新查询进入等待队列
- 等待超时会抛出 `等待查询槽位超时`
- `QueryExecutor` 的查询类方法会在执行前 `acquireQuerySlot()`，结束后 `releaseQuerySlot()`

### 2.3 一个容易踩坑的事实

`setQueryTimeout()` 不只影响 SQL 执行超时。

- 在 `QuerySlotGate` 中，它还是“等待槽位”的超时值
- 在 `QueryExecutor` 中，它还是“查询结果返回”的超时值

这意味着你把超时设得过小，先炸的未必是慢 SQL，也可能是排队中的正常请求。

## 3. 参数建议

| 场景 | 连接池建议 | 查询槽位建议 | 说明 |
| --- | --- | --- | --- |
| 单线程或低并发后台任务 | `minSize=1, maxSize=1` | `maxConcurrentQueries=1~2` | 避免人为制造排队 |
| 普通列表查询 + 少量写入 | `minSize=2, maxSize=4` | `maxConcurrentQueries` 与 `maxSize` 持平或略高 1 | 先看 `pendingRequests` 再决定是否扩容 |
| 批量读取较多 | `minSize=2, maxSize=4~6` | `maxConcurrentQueries=maxSize` | 不要让槽位远大于连接数 |
| 超时排队频繁 | 优先增大 `maxSize` 或降低请求并发 | 再考虑上调 `queryTimeoutMs` | 只加超时是在掩盖排队 |

## 4. 正确示例

### 4.1 正确的初始化方式

```ts
import { DatabaseConfig, DatabaseManager } from 'ocorm'

const config = new DatabaseConfig('app.db')
  .setConnectionPool({
    enabled: true,
    minSize: 2,
    maxSize: 4,
    acquireTimeoutMs: 1500
  })
  .setMaxConcurrentQueries(4)
  .setQueryTimeout(1200)

await DatabaseManager.getInstance().initialize(context, config)

const metrics = DatabaseManager.getInstance().getPoolMetrics()
console.info(`poolEnabled=${DatabaseManager.getInstance().isPoolEnabled()}`)
console.info(JSON.stringify(metrics))
```

为什么这样配：

- `maxConcurrentQueries` 不高于 `maxSize`，排队点统一落在连接池之外的概率更低
- `acquireTimeoutMs` 和 `queryTimeoutMs` 都保留了失败边界，不会无限等待
- `getPoolMetrics()` 可以直接验证池是否进入了长期排队

### 4.2 正确的租约使用方式

手动租约只有一个铁律：`finally` 中释放。`StoreLease.release()` 是幂等的，但“幂等”不等于“可以不释放”。

```ts
import { DatabaseManager } from 'ocorm'

const manager = DatabaseManager.getInstance()
const lease = await manager.acquireStoreLease()

try {
  await lease.store.executeSql(
    'UPDATE users SET user_name = ? WHERE id = ?',
    ['worker-13', 1]
  )
} finally {
  lease.release()
}
```

如果只是包一段逻辑，优先用 `withStoreLease()`，因为它自带释放。

```ts
import { DatabaseManager } from 'ocorm'

await DatabaseManager.getInstance().withStoreLease(async (lease) => {
  await lease.store.executeSql(
    'INSERT INTO users (id, user_name) VALUES (?, ?)',
    [7, 'pooled-write']
  )
})
```

## 5. 误用示例

下面这类配置表面上“并发很大”，实际是在制造排队和超时。

```ts
import { DatabaseConfig } from 'ocorm'

const badConfig = new DatabaseConfig('app.db')
  .setConnectionPool({
    enabled: true,
    minSize: 1,
    maxSize: 1,
    acquireTimeoutMs: 0
  })
  .setMaxConcurrentQueries(16)
  .setQueryTimeout(50)
```

问题有三个：

- `maxSize=1`，但允许 16 个查询同时进入门控，真正瓶颈仍然只有一个连接
- `acquireTimeoutMs=0` 表示池满时立即失败，不会等待任何连接归还
- `queryTimeoutMs=50` 同时限制槽位等待和查询执行，轻微抖动就会报超时

还有一种误用是手动获取租约但不释放，症状是 `activeLeases` 持续不回落、`pendingRequests` 增长、随后出现 `获取连接超时`。

```ts
import { DatabaseManager } from 'ocorm'

const lease = await DatabaseManager.getInstance().acquireStoreLease()
await lease.store.executeSql('DELETE FROM users WHERE id = ?', [1])

// 误用：没有 release()
// 连接不会回到池中，后续请求可能永久等到超时
```

## 6. 观测指标

上线前至少盯住下面 6 个数字：

- `activeLeases`
- `idleStores`
- `pendingRequests`
- `timeoutCount`
- `getActiveQuerySlots()`
- 错误消息中是否出现 `获取连接超时` 或 `等待查询槽位超时`

可以直接做一个最小观测函数：

```ts
import { DatabaseManager } from 'ocorm'

function dumpPoolAndGate(tag: string): void {
  const manager = DatabaseManager.getInstance()
  const metrics = manager.getPoolMetrics()

  console.info(`[${tag}] activeQuerySlots=${manager.getActiveQuerySlots()}`)
  console.info(`[${tag}] pool=${JSON.stringify(metrics)}`)
}
```

判读原则很简单：

- `activeLeases` 长期接近 `maxSize`：连接池已打满
- `pendingRequests > 0` 且持续增长：池容量不足或租约未归还
- `timeoutCount` 持续增加：先查是否泄漏租约，再查是否把 `maxConcurrentQueries` 设得明显大于 `maxSize`
- `getPoolMetrics() === null`：连接池根本没开，此时只存在单例连接

## 7. 复现实验

`EntryConnectionPoolSuite.ets` 已经覆盖了默认值、归一化、单例回退和租约释放行为。手工复现时建议只做一件事：构造“池容量小于并发请求数”的场景，确认指标是否符合预期。

```ts
import { DatabaseConfig, DatabaseManager } from 'ocorm'

const config = new DatabaseConfig('pool-lab.db')
  .setConnectionPool({
    enabled: true,
    minSize: 1,
    maxSize: 2,
    acquireTimeoutMs: 800
  })
  .setMaxConcurrentQueries(2)
  .setQueryTimeout(1200)

await DatabaseManager.getInstance().initialize(context, config)

const jobs = [1, 2, 3, 4].map((id) => {
  return DatabaseManager.getInstance().withStoreLease(async (lease) => {
    await lease.store.querySql('SELECT 1')
    await new Promise<void>((resolve) => setTimeout(resolve, 300))
    console.info(`job ${id} done`)
  })
})

await Promise.allSettled(jobs)
dumpPoolAndGate('after-jobs')
```

预期现象：

- 前两个任务先拿到连接
- 后两个任务进入等待
- 如果等待时间小于 `acquireTimeoutMs`，它们会在前序任务释放后继续执行
- 如果你把 `acquireTimeoutMs` 改到 100ms，后续任务更容易直接失败

## 8. 调优顺序

不要同时乱拧所有参数。按这个顺序来：

1. 先确认是否真的开启了连接池，`getPoolMetrics()` 不能是 `null`
2. 再看 `activeLeases` 是否长期打满
3. 如果打满，先调 `maxSize`
4. 如果没打满但仍然频繁超时，检查 `queryTimeoutMs` 是否过小
5. 如果 `pendingRequests` 上升很快，检查业务代码是否忘记释放租约，或是否把 `maxConcurrentQueries` 设得明显大于 `maxSize`

## 9. 当前实现边界

- `EntryConnectionPoolSuite` 验证了 `setConnectionPool()` 的归一化，默认池配置为关闭，默认值表现为 `minSize=1`、`maxSize=1`、`acquireTimeoutMs=3000`
- `FixedStorePool` 当前没有实现基于 `idleTimeoutMs` 的空闲连接回收
- `FixedStorePool` 当前没有实现基于 `connectionPool.healthCheckIntervalMs` 的池内连接健康检查
- `DatabaseConfig.healthCheckIntervalMs` 是 `DatabaseManager` 级别的 `ping()` 定时检查，不等于连接池健康巡检
