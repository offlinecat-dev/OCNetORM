# 05-事务守卫-嵌套-跨仓库-未绑定写入

> 状态：已完成
> 适用版本：ocorm 3.x（当前仓库实现）
> 最后更新：2026-03-27

## 1. 目标
- 说明 `RepositoryTransactionGuardSupport.ets` 如何阻止危险写入路径。
- 说明三类高频错误：同仓库嵌套事务、跨仓库嵌套事务、未绑定事务上下文写入。
- 给出正确复用 `txRepo` 的代码模式。

## 2. 背景与适用范围
仓储层不是无脑把 SQL 发到连接上。写入前会先走事务守卫：
- `ensureWritePathAllowed(operation)`
- `ensureNoNestedTransaction(entryName)`

守卫的目的是避免“外层事务正在跑，内层却偷偷换连接或换仓储继续写”，从而破坏事务边界。

## 3. 守卫触发条件
- 当前仓储处在严格事务回调里：允许该 `txRepo` 继续写入
- 连接池模式下存在活动事务，但当前仓储不是绑定事务仓储：拒绝
- 当前连接级事务已激活，但你试图从事务外再开事务或写入：拒绝

```text
strict tx repo -> allow
pooled active tx + unbound repo -> block
same connection has active tx + outer repo write -> block
same repo nested transaction -> block
cross repo nested transaction -> block
```

## 4. 正确示例
正确做法是在事务回调里只使用回调提供的 `txRepo`，不要回头再用外层仓储。

```arkts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const userRepo = new Repository('User')

await userRepo.transaction(async (txRepo) => {
  await txRepo.save(makeUser(1, 'inside-tx'))
  await txRepo.rawExecute('UPDATE user SET name = ? WHERE id = ?', ['done', 1])
})
```

当前公开 API 不提供“从 `txRepo` 再派生另一个实体仓储”的入口。需要跨实体原子写入时，应优先复用当前 `txRepo` 已暴露的关系/级联能力；不要假设可以在回调里再 `new Repository(...)` 并共享同一事务。

```arkts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makePost(id: number, title: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('title', title)
  return EntityData.from('Post', input)
}

const postRepo = new Repository('Post')

await postRepo.transaction(async (txRepo) => {
  await txRepo.save(makePost(10, 'A'))
  await txRepo.attach(10, 3, 'tags')
})
```

## 5. 误用示例
同一仓库嵌套事务会直接失败，不存在“自动变 SAVEPOINT”的降级。

```arkts
import { Repository } from 'ocorm'

const repo = new Repository('User')

await repo.transaction(async (txRepo) => {
  // 误用：同一 Repository 嵌套开事务
  await txRepo.transaction(async (_nestedRepo) => {
    console.info('never here')
  })
})
```

外层事务里再用外部仓储写入，会被当成“未绑定事务上下文写入”。

```arkts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const outerRepo = new Repository('User')

await outerRepo.transaction(async (txRepo) => {
  await txRepo.save(makeUser(20, 'ok'))

  // 误用：继续用外层 repo 写入
  await outerRepo.save(makeUser(21, 'blocked'))
})
```

跨仓库嵌套事务同样不允许。

```arkts
import { Repository } from 'ocorm'

const userRepo = new Repository('User')
const postRepo = new Repository('Post')

await userRepo.transaction(async (_txRepo) => {
  // 误用：另一个 Repository 再次开启事务
  await postRepo.transaction(async (_postTxRepo) => {
    console.info('never here')
  })
})
```

## 6. 失败语义与抛错说明
这类错误都会抛 `TransactionRollbackError`，而不是返回 `success = false`。它们的定位关键词分别是：
- 未绑定写入：连接池活动事务 / 连接级事务已激活
- 同仓库嵌套：同一 `Repository` 嵌套调用 `transaction` / `transactionWithOptions`
- 跨仓库嵌套：外层事务已开，另一个仓储再开事务

```arkts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const repo = new Repository('User')

try {
  await repo.transaction(async (txRepo) => {
    await repo.removeById(1)
    await txRepo.save(makeUser(2, 'B'))
  })
} catch (error) {
  console.error('事务守卫已拦截:', error)
}
```

## 7. 排障与验收
- 在事务回调里，一律检查自己调用的是不是 `txRepo`
- 连接池模式下出现“禁止未绑定事务上下文写入”时，优先排查是不是漏用了外层仓储
- 如果你想要 SAVEPOINT 语义，应看关系写入和级联实现，不要指望仓储事务自动嵌套

```text
排查顺序:
1. 是否在 transaction/transactionWithOptions 回调里
2. 当前调用对象是不是回调参数 txRepo
3. 是否在另一个 Repository 上再次调用 transaction
4. 是否开启了连接池并存在活动事务
```
