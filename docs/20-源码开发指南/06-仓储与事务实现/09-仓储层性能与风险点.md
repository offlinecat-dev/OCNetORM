# 09-仓储层性能与风险点

> 状态：已完成
> 适用版本：ocorm 3.x（当前仓库实现）
> 最后更新：2026-03-27

## 1. 目标
- 汇总 `Repository` 层最容易被误用的性能热点。
- 说明哪些 API 天生是顺序路径，哪些 API 才是批量路径。
- 给出性能优化时必须同时看的风险边界。

## 2. 高风险热点
- `saveAll`：顺序逐条保存，不是数据库级批处理
- `attach`：先查重再插入，重复调用有额外查询成本
- `sync`：先删后逐条插入，目标集合越大越慢
- `rawExecute`：成功后清空整个查询缓存
- `transactionWithOptions(retries > 0)`：非幂等写入可能被重放

```text
优先级从高到低:
1. 大批量导入误用 saveAll
2. 高频关系同步误用 sync
3. 原生写 SQL 高频触发 queryCache.clear()
4. 非幂等事务配置 retries
5. 事务外误用外层 repo 触发守卫回滚
```

## 3. 正确示例
大批量导入优先 `batchInsert`，不要循环 `save` 或 `saveAll`。

```arkts
import { BatchInsertOptions, EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const repo = new Repository('User')

const result = await repo.batchInsert(
  new Array(500).fill(0).map((_, index) => makeUser(index + 1, `User-${index + 1}`)),
  BatchInsertOptions.createDefault()
)

console.info(result.insertedCount)
```

关系同步如果目标集合不大且你明确要“全量覆盖”，再使用 `sync`。

```arkts
import { Repository } from 'ocorm'

const repo = new Repository('Post')
await repo.sync(10, [1, 2, 3], 'tags')
```

## 4. 误用示例
下面这段代码在功能上可行，但吞吐会非常差，因为每条都要走完整 `save` 路径。

```arkts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const repo = new Repository('User')

await repo.saveAll(
  new Array(1000).fill(0).map((_, index) => makeUser(index + 1, `User-${index + 1}`))
)
```

另一个误用是对高频、细粒度更新都走 `rawExecute`，结果把缓存一遍遍全清。

```arkts
import { Repository } from 'ocorm'

const repo = new Repository('User')

for (let i = 0; i < 100; i++) {
  await repo.rawExecute('UPDATE user SET score = score + ? WHERE id = ?', [1, i + 1])
}
```

## 5. 风险与副作用
`batchInsert` 追求吞吐，但在 `createFast()` 下会关闭验证和钩子。性能更高，不代表业务语义更安全。

```arkts
import { BatchInsertOptions } from 'ocorm'

const fast = BatchInsertOptions.createFast()
console.info(fast.useTransaction, fast.executeHooks, fast.executeValidation)
```

重试事务也有代价。当前实现会在失败后重新跑 `action`，如果你的事务体里包含外部调用、幂等性差的副作用，风险会被放大。

```arkts
import { TransactionOptions } from 'ocorm'

const options = TransactionOptions.withRetry(2, 100)
console.info(options.retries, options.retryDelay)
```

## 6. 调优规则
- “大量新增”优先 `batchInsert`
- “复杂查询”优先 `QueryBuilder` 或 `QueryExecutor`，不要用 `saveAll` 或原生 SQL 兜底
- 高频关系改动要区分 `attach` 的幂等追加和 `sync` 的全量覆盖成本
- 原生写 SQL 要有意识地控制频率，因为它会清空缓存
- 事务重试只用于幂等逻辑

## 7. 验收与观察
可以用以下指标判断仓储层是否被错误使用：

```text
1. 导入路径是否大量出现 save/saveAll
2. 关系更新是否频繁执行 delete + insert 全覆盖
3. 慢查询/慢写日志是否集中出现在 RAW EXECUTE
4. 是否频繁出现事务守卫报错导致回滚
5. query cache 命中率是否被高频 clear 拖低
```
