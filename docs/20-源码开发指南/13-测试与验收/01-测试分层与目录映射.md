# 01-测试分层与目录映射

> 状态：已完成
> 适用版本：ocorm 3.x（以仓库当前测试目录为准）
> 最后更新：2026-03-28

## 1. 目标

本文只回答一个问题：测试代码应该放在哪里，以及每一层到底验证什么。

本仓库当前存在 5 层真实测试资产：

1. `OCORM/src/test/*`：OCORM 库级 Hypium 单测。
2. `entry/src/test/*`：entry 模块侧 Hypium 回归/同步测试。
3. `entry/src/main/ets/testcore/*` + `entry/src/main/ets/suites/*`：可编排 Suite 运行层。
4. `OCORM/src/ohosTest/*`：OCORM 模块设备侧 `ohosTest` 测试。
5. `entry/src/ohosTest/*`：entry 模块设备侧 `ohosTest` 测试。

## 3. 入口关系

### 3.1 `OCORM/src/test`

- `OCORM/src/test/List.test.ets` 当前只导入 `Smoke.test.ets`。
- `OCORM/src/test/Stage*/*.test.ets` 才是分阶段能力测试主体。
- 结论：不要想当然地认为 `OCORM/src/test/List.test.ets` 已经覆盖所有 Stage 文件；是否执行到具体 Stage，必须以当前入口和运行配置为准。

### 3.2 `entry/src/test`

- `entry/src/test/List.test.ets` 当前手工导入以下测试：
  - `LocalUnit.test.ets`
  - `EntrySuiteProfile.test.ets`
  - `EntryBatchInsertTransactionRegression.test.ets`
  - `EntryOrmReviewReport20Fix.test.ets`
  - `EntryRelationTimeout.test.ets`
  - `EntryOrmPerformanceBatch.test.ets`
  - `EntryOrmConcurrencyFix.test.ets`
  - `EntryOrmObservabilityFix.test.ets`
  - `EntryOrmSecurityFix.test.ets`
- 同目录下还存在未被该 `List.test.ets` 聚合的 Hypium 文件，例如：
  - `EntryConnectionPool.test.ets`
  - `EntryOrmCoreFix.test.ets`
  - `EntryOrmCriticalFix.test.ets`
  - `EntryOrmReviewFix.test.ets`
  - `EntryOrmReviewReport19Fix.test.ets`
  - `EntryOrmSubQueryFix.test.ets`
  - `EntryOrmTools.test.ets`

### 3.3 `entry/src/main/ets/suites`

- `entry/src/main/ets/suites/index.ets` 通过 `getAllSuites()` 注册 Suite。
- 这里同时注册了核心套件与 entry 同步套件，例如：
  - `CoreSuite`
  - `QuerySuite`
  - `RepositorySuite`
  - `TransactionSuite`
  - `BoundarySuite`
  - `EntryConnectionPoolSuite`
  - `EntryOrmSecurityFixSuite`
  - `EntryOrmSubQueryFixSuite`
  - `EntryListSuite`
- `PerformanceSuite` 是否加入由 `profile` / `includePerformance` 决定，不是默认恒定开启。

### 3.4 `entry/src/ohosTest`

- `entry/src/ohosTest/ets/test/List.test.ets` 当前只聚合 `Ability.test.ets`。
- 这一层是设备侧测试入口，和 `entry/src/main/ets/suites/index.ets` 的 Suite 编排是两条线。

### 3.5 `OCORM/src/ohosTest`

- `OCORM/src/ohosTest/ets/test/List.test.ets` 当前也只聚合 `Ability.test.ets`。
- 这一层属于库模块的设备测试入口，与 `OCORM/src/test/*` 的本地 Hypium 单测是两条执行线。

## 4. 正确示例

下面这个写法符合当前仓库的真实分层：Hypium 列表文件只负责聚合同层级的 `.test.ets` 文件，不跨层调用 Suite 类。

```ts
import localUnitTest from './LocalUnit.test';
import entrySuiteProfileTest from './EntrySuiteProfile.test';
import entryBatchInsertTransactionRegressionTest from './EntryBatchInsertTransactionRegression.test';
import entryOrmReviewReport20FixTest from './EntryOrmReviewReport20Fix.test';
import entryRelationTimeoutTest from './EntryRelationTimeout.test';
import entryOrmPerformanceBatchTest from './EntryOrmPerformanceBatch.test';
import entryOrmConcurrencyFixTest from './EntryOrmConcurrencyFix.test';
import entryOrmObservabilityFixTest from './EntryOrmObservabilityFix.test';
import entryOrmSecurityFixTest from './EntryOrmSecurityFix.test';

export default function testsuite() {
  localUnitTest();
  entrySuiteProfileTest();
  entryBatchInsertTransactionRegressionTest();
  entryOrmReviewReport20FixTest();
  entryRelationTimeoutTest();
  entryOrmPerformanceBatchTest();
  entryOrmConcurrencyFixTest();
  entryOrmObservabilityFixTest();
  entryOrmSecurityFixTest();
}
```

同理，Suite 层也有自己的真实入口，不应混入 `.test.ets` 聚合逻辑：

```ts
export function getAllSuites(options?: SuiteSelectionOptions): TestSuite[] {
  const suites: TestSuite[] = []

  suites.push(new CoreSuite())
  suites.push(new QuerySuite())
  suites.push(new RepositorySuite())
  suites.push(new TransactionSuite())
  suites.push(new BoundarySuite())
  suites.push(new EntryConnectionPoolSuite())
  suites.push(new EntryOrmSecurityFixSuite())
  suites.push(new EntryOrmSubQueryFixSuite())
  suites.push(new EntryListSuite())

  return suites
}
```

## 5. 误用示例

下面这种写法是错的：它把 `entry/src/main/ets/suites/*` 当成 Hypium `List.test.ets` 来直接调用，破坏了分层和注册边界。

```ts
import { EntryOrmSecurityFixSuite } from '../main/ets/suites/EntryOrmSecurityFixSuite'

export default function testsuite() {
  const suite = new EntryOrmSecurityFixSuite()
  suite.registerCases()
  // 错误点 1：Hypium 列表入口不应该直接操纵 Suite 类
  // 错误点 2：TestRunner.registerSuite() 才是 registerCases() 的真实调用点
  // 错误点 3：这样做不会自动接入 TestContext / TestReporter / 过滤配置
}
```

另一个常见误用是把设备测试和应用内 Suite 混在一起：

```ts
import abilityTest from '../../ohosTest/ets/test/Ability.test'
import { getAllSuites } from '../main/ets/suites/index'

export default function testsuite() {
  abilityTest()
  getAllSuites()
  // 错误点：ohosTest 与 suites 属于不同运行线，不应在同一入口里硬拼
}
```

## 6. 验收命令与检查清单

先做静态核对，再做构建/测试，不要跳步。

```powershell

# 3. 运行仓库已有测试入口

```

```text
[验收检查清单]
1. 文档中提到的目录和文件，仓库内必须真实存在。
2. 明确区分 Hypium `.test.ets` 入口 与 Suite 运行层。
3. 明确区分 `OCORM/src/test`、`OCORM/src/ohosTest`、`entry/src/test`、`entry/src/ohosTest` 四条入口线。

5. 不得声称 `OCORM/src/test/List.test.ets` 已经聚合全部 Stage 测试，除非先重新核对源码。
```

## 7. 结论

当前仓库的测试分层不是“一个目录跑全部”，而是五条职责线并存：

1. `OCORM/src/test/*` 负责库级单测。
2. `entry/src/test/*` 负责 entry 侧 Hypium 回归。
3. `entry/src/main/ets/testcore/*` + `entry/src/main/ets/suites/*` 负责可编排 Suite 运行。
4. `OCORM/src/ohosTest/*` 负责 OCORM 设备测试。
5. `entry/src/ohosTest/*` 负责 entry 设备测试。

写新测试前先选层，再选入口，最后再决定是否需要同步到 Suite。顺序反了，就会开始堆测试债。
