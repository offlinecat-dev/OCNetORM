## 1. 这套文档解决什么问题

这不是宣传材料，也不是空泛设计稿。

这套文档只做三件事：

1. 对齐当前仓库真实代码、真实导出面、真实测试入口。
2. 给开发者提供从建模、查询、事务、迁移到排障的稳定查阅路径。
3. 让新增功能、修复项和审查结论都有落点，不再把知识散落在提交记录和聊天里。

当前目录共维护 `19` 个主题目录，当前 Markdown 文档总数为 `104`（含本索引与目录级 `README.md`）。

## 2. 命名与维护规则

所有新增文档都按这一套规则继续写，不再回到混乱命名：

- 目录命名：`NN-主题`
- 文件命名：`NN-主题.md`
- 目录级入口：仅 `README.md` 保留给目录总览或索引页
- API 速查目录允许在标题中保留 `API`、`Schema`、`SQL` 这类技术词，其余尽量中文化
- 交叉引用全部使用相对路径，禁止写“见某文档”但不给链接

```text
正确示例
00-导航与约定/
  README.md
  01-文档阅读路径.md

18-API速查/
  README.md
  01-核心入口-错误与类型API.md
  02-查询-关系与缓存API.md
```

```text
错误示例
18-API速查/
  core-api.md
  query-notes-final.md
  临时草稿1.md
```

## 5. 交叉引用约定

章节之间的跳转，统一按“从接口到实现、从能力到约束、从改动到验收”的方向组织：

```text
核心建模 -> 数据库与连接层 -> 查询体系 -> 仓储与事务 -> Schema与迁移
                                     \-> 验证与钩子 -> 错误与日志
查询 / 事务 / Schema 改动 -> 安全基线 -> 性能与调优 -> 测试与验收 -> 治理与审查归档
```

维护文档时至少做这三步检查：

```powershell
# 1. 清理草稿和占位符
$draftTokens = @('状态：草' + '稿', '待补' + '充', 'TO' + 'DO', 'TB' + 'D', '骨架已创' + '建')
rg -n ($draftTokens -join '|') OCORM/docs/开发者文档

# 2. 检查 API 速查目录命名是否仍然统一
Get-ChildItem 'OCORM/docs/开发者文档/18-API速查' -Name

# 3. 统计文档总数，避免索引失真
(Get-ChildItem 'OCORM/docs/开发者文档' -Recurse -Filter *.md | Measure-Object).Count
```
