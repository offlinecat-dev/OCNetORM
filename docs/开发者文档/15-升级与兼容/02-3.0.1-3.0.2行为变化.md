# 02-3.0.1-3.0.2行为变化

> 状态：已完成
> 适用版本：从 3.0.1 升级到当前仓库 3.0.2
> 最后更新：2026-03-28

## 1. 文档边界
本文只使用仓库内可验证材料：
- `OCORM/CHANGELOG.md`
- `OCORM/README.md`
- `OCORM/src/main/ets/**` 当前 3.0.2 实现
- `OCORM/docs/开发者文档/00-导航与约定`、`05-查询体系`、`06-仓储与事务` 的现有文档

说明口径分两类：
- `3.0.2` 当前行为：可直接由当前源码证明
- `3.0.1` 历史行为：若当前仓库没有保留旧实现，仅根据 `CHANGELOG.md` 描述；涉及代码写法时明确标注“从当前仓库可推断”

## 2. 一句话结论
`3.0.2` 不是大范围 API 重写，主要变化集中在两个地方：
- `rawQuery` 从 `3.0.1` 的“只读校验”进一步收紧为“默认强制参数化 + 占位符数量严格匹配”
- 测试桥接入口从 `DatabaseManager` 主类迁到 `database/testing/DatabaseManagerTestHooks`

其余大量改动以内部职责拆分和测试修复为主，对包入口 `import { ... } from 'ocorm'` 没有新增 breaking 导入方式变化。

## 3. 3.0.1 已经存在、3.0.2 继续保留的行为
下面这些行为不是 `3.0.2` 新引入，而是 `3.0.1` 已明确建立，`3.0.2` 继续延续：
- 实体查询路径拒绝 `selectRaw/groupBy/having`
- 需要原生/聚合结果时改用 `getRaw()` 或 `aggregate()`
- `rawQuery` 的安全校验会剔除注释与字符串内容，避免误判和绕过

这些点可由 `CHANGELOG.md` 的 `3.0.1` 条目和当前 `QueryExecutor.ets`、`RawSqlGuards.ets` 的实现共同支撑。

## 4. 3.0.1 -> 3.0.2 的核心变化
### 4.1 `rawQuery` 从“只读校验”升级为“默认强制参数化”
`CHANGELOG.md` 对 `3.0.2` 的描述非常直接：
- `rawQuery` 改为默认强制参数化：无占位符或参数数量不匹配时直接拒绝执行（fail-fast）

当前 `Repository.ets` 也能直接证明这一点：
- `rawQuery()` 内部会先走 `validateRawQuerySql(sql)`
- 然后统计 `?` 占位符数量
- `placeholderCount <= 0` 或 `placeholderCount !== args.length` 都会拒绝

因此，`3.0.1` 到 `3.0.2` 的最重要升级动作，就是把历史上仍然存在的“无参数占位符 SELECT”统一改掉。

### 4.2 测试专用 API 迁移到 `DatabaseManagerTestHooks`
`CHANGELOG.md` 在 `3.0.2` 明确写到：
- `DatabaseManager` 测试注入 API 迁移到测试专用入口 `database/testing/DatabaseManagerTestHooks`
- 主类移除 `getTestState/setTestState/applyRuntimeConfigForTest/resetInstance`

当前源码也可直接证明：
- `OCORM/src/main/ets/database/testing/DatabaseManagerTestHooks.ets` 存在并暴露 `getState/setState/applyRuntimeConfig/resetInstance`
- `DatabaseManager.ets` 保留的是 `getTestingState/setTestingState/applyRuntimeConfigForTesting/resetInstanceForTesting` 供测试桥接调用，而不是旧公开面

这个变化主要影响测试代码，不影响正常业务导入入口。

### 4.3 内部模块拆分不等于外部 API 变化
`3.0.2` 把若干守卫和执行逻辑拆到了独立模块：
- `rawsql/RawSqlGuards`
- `query/QueryBuilderGuards`
- 一系列 `QueryExecutor*Support` / `RelationLoader*Support`

这类变化可由 `CHANGELOG.md` 和当前源码目录直接证明，但它们本质上是内部职责下沉，不意味着调用方必须改 `import { ... } from 'ocorm'`。

## 5. 升级前后代码对照
### 5.1 `rawQuery` 调用方式
下面的“升级前”示例不是对 `3.0.1` 源码逐行复原，而是根据 `3.0.2 CHANGELOG` 可直接推导出的高风险旧写法。

```ts
// 升级前：从当前仓库可推断，3.0.1 时代仍可能存在这种无占位符调用
const rows = await repo.rawQuery('SELECT id, name FROM user')
```

```ts
// 升级后：3.0.2 要求参数化，即使没有动态值，也要避免继续依赖旧写法
const rows = await repo.rawQuery(
  'SELECT id, name FROM user WHERE is_active = ?',
  [1]
)
```

### 5.2 聚合查询入口
这部分变化锚点在 `3.0.1`，`3.0.2` 继续延续，不应再回退到旧混用方式。

```ts
// 升级前：从 3.0.1 起就应避免的写法
const rows = await new QueryExecutor(
  repo.createQueryBuilder()
    .selectRaw(['status', 'COUNT(*) AS total'])
    .groupBy('status')
    .having('COUNT(*) > ?', [1])
).get()
```

```ts
// 升级后：维持 3.0.1/3.0.2 统一语义
const rows = await new QueryExecutor(
  repo.createQueryBuilder()
    .selectRaw(['status', 'COUNT(*) AS total'])
    .groupBy('status')
    .having('COUNT(*) > ?', [1])
).getRaw()
```

### 5.3 测试桥接入口
该差异由 `3.0.2` 直接引入。

```ts
// 升级前：3.0.2 CHANGELOG 标明这是旧公开测试入口
DatabaseManager.resetInstanceForTesting()
manager.applyRuntimeConfigForTesting(config)
```

```ts
// 升级后：改走测试专用桥接类
DatabaseManagerTestHooks.resetInstance()
DatabaseManagerTestHooks.applyRuntimeConfig(manager, config)
```

## 6. 误用示例
这些写法在升到当前 `3.0.2` 后都应视为不兼容或高风险。

```ts
// 误用 1：rawQuery 没有占位符
await repo.rawQuery('SELECT * FROM user')

// 误用 2：实体模式继续混用 selectRaw/groupBy/having
await new QueryExecutor(
  repo.createQueryBuilder().selectRaw(['COUNT(*) AS total'])
).get()

// 误用 3：测试代码继续依赖旧桥接路径
DatabaseManager.resetInstanceForTesting()
```

对应结果：
- 误用 1：`3.0.2` 当前实现会因占位符数量校验失败而拒绝
- 误用 2：当前 `QueryExecutor.ets` 会直接抛实体模式限制错误
- 误用 3：升到 `3.0.2` 时需要迁移到 `DatabaseManagerTestHooks`

## 7. 兼容性检查清单
从 `3.0.1` 升到 `3.0.2`，至少核对下面这些项：

- [ ] 所有 `rawQuery(...)` 调用都包含 `?` 占位符，且参数数量与占位符一致
- [ ] `rawQuerySafe(...)` 与 `rawExecute(...)` 也没有继续保留字符串拼接 SQL
- [ ] 查询代码没有把 `selectRaw/groupBy/having` 和 `get/getPaginated/count` 混用
- [ ] 测试代码已从旧 `DatabaseManager` 测试公开面迁移到 `DatabaseManagerTestHooks`
- [ ] 业务代码没有因为内部模块拆分而改成引用 `*Support` 或 `rawsql/*` 的内部实现文件

```bash
rg -n "rawQuery\(|rawQuerySafe\(|rawExecute\(|selectRaw\(|groupBy\(|having\(|getRaw\(|aggregate\(|DatabaseManagerTestHooks|resetInstanceForTesting|applyRuntimeConfigForTesting" OCORM entry
```

## 8. 最小回归建议
推荐至少跑三类回归：
- 原生 SQL 只读查询回归：确认 `rawQuery/rawQuerySafe` 的占位符规则全量通过
- 聚合查询回归：确认统计接口全部走 `getRaw()` 或 `aggregate()`
- 测试基础设施回归：确认依赖 `DatabaseManager` 测试状态的单测或集成测不再调用旧桥接入口

```bash
ohpm install
hvigor test
```

## 9. 当前不应夸大的点
以下变更可以在文档里说明，但不应被写成“调用方式已变”：
- `RawSqlGuards`、`QueryBuilderGuards`、`QueryExecutor*Support` 的模块拆分主要是内部重构
- `release` 混淆开启、`HAR_VERSION=3.0.2` 对齐、`ohosTest` 设备矩阵补齐属于构建与测试治理变化，不是业务 API 迁移主线
