# 04-兼容性声明模板

> 状态：已完成
> 用途：发布说明、升级公告、PR/变更审查说明
> 最后更新：2026-03-28

## 1. 目标
- 给当前仓库提供统一的兼容性声明写法。
- 让“兼容”“breaking”“需迁移”三种结论可复用、可审计。
- 避免发布说明里出现没有证据支撑的承诺。

## 2. 使用原则
- 只声明仓库里能证明的行为差异
- 若历史实现不可直接验证，必须写“从当前仓库可推断”
- 一份声明至少包含：版本范围、受影响能力、升级动作、验证建议

## 3. 标准模板

```md
# 兼容性声明

> 版本范围：从 `<fromVersion>` 到 `<toVersion>`
> 结论类型：兼容 / 兼容但需迁移 / breaking
> 证据来源：CHANGELOG / 当前源码 / 测试套件 / 开发者文档

## 1. 影响范围
- 模块：
- 关键能力：

## 2. 当前可证明行为
- ...

## 3. 升级动作
- ...

## 4. 风险与限制
- ...

## 5. 最小回归建议
- ...
```

## 4. 兼容声明示例

```md
# 兼容性声明

> 版本范围：`3.0.1 -> 3.0.2`
> 结论类型：兼容但需迁移
> 证据来源：`OCORM/CHANGELOG.md`、`Repository.ets`、`RawSqlGuards.ets`

## 1. 影响范围
- 模块：原生 SQL 查询路径
- 关键能力：`rawQuery/rawQuerySafe`

## 2. 当前可证明行为
- `rawQuery` 默认强制参数化
- 无占位符或参数不匹配时 fail-fast

## 3. 升级动作
- 把字符串拼接查询统一替换成 `?` 占位符

## 4. 风险与限制
- 若仍保留旧宽松查询写法，升级后会直接抛错
```

## 5. 代码评审附带模板
如果是在 PR 或审查报告里附带兼容性结论，推荐最小字段如下：

```text
兼容结论:
- 类型: compatible / compatible-with-migration / breaking
- 版本跨度: x.y.z -> a.b.c
- 受影响入口: API / 测试桥接 / 构建配置 / 文档
- 证据路径: 至少 2 条
- 必需升级动作: 1~3 条
- 不可证事项: 如无旧实现源码，必须写明
```

## 6. 误用示例
下面这种写法不允许：

```md
结论：完全兼容，无需任何调整
```

如果仓库已经显示存在参数化收紧、事务守卫收紧、测试桥接迁移，这种一句话“完全兼容”就是失真。正确写法应该把限制写出来。
