# 05-验证与钩子协同顺序

> 状态：完成
> 适用版本：ocorm 3.x（基于当前源码）
> 最后更新：2026-03-27

## 1. 目标
这份文档只解决一个问题：验证系统和生命周期钩子在真实仓库调用链里到底谁先谁后。
结论先说死：写入路径上，字段验证先于 `before*` 钩子；读取路径、删除路径、恢复路径默认不走 `EntityValidator`，只走对应钩子。
如果你把“在 `beforeSave` 里修补非法字段”当成默认保存链路的一部分，那是错误预期。

## 2. 背景与适用范围
适用于以下场景：
- 需要梳理单条保存时 `EntityValidator` 与 `HooksProcessor` 的先后顺序
- 需要确认批量插入是否和单条保存拥有相同生命周期
- 需要排查“为什么钩子没执行”“为什么只报验证错误没有进入钩子”“为什么查询后只触发 `afterLoad`”

本文只讨论真实源码中已经存在的协作关系：
- 规则来源：`ValidationMetadataStorage`
- 规则执行：`EntityValidator`
- 生命周期执行：`HooksProcessor`
- 错误出口：`ValidationError` / `HookExecutionError`

## 3. 前置条件
- 调用方已经为实体声明过验证规则，并已准备好实体数据对象
- 调用方已经为实体级或全局级生命周期注册钩子
- 调用方理解 `EntityValidator.validateOrThrow` 与 `HooksProcessor.executeXxx` 都是显式调用，不会彼此自动代替
- 如果你在业务层手动编排流程，必须自己决定是在验证通过后再执行钩子，还是先执行钩子再做其他逻辑

## 4. 关键概念与约束
### 4.1 单条保存链路的真实顺序
单条保存场景里，验证一定先于 `beforeSave`：

```text
save(entityData)
  -> EntityValidator.validateOrThrow(entityName, entityData)
  -> 额外唯一约束校验
  -> HooksProcessor.executeBeforeSave(entityName, entityData)
  -> HooksProcessor.executeBeforeInsert(...) 或 executeBeforeUpdate(...)
  -> 数据库写入
  -> HooksProcessor.executeAfterInsert(...) 或 executeAfterUpdate(...)
  -> HooksProcessor.executeAfterSave(entityName, entityData)
```

直接推论：
- `ValidationError` 一旦抛出，后续 `beforeSave`、`beforeInsert`、`beforeUpdate` 都不会执行
- `HookExecutionError` 只会出现在验证已经通过、并且某个钩子开始执行之后

### 4.2 默认保存不会传 `groups` 或 `uniqueChecker`
真实保存链路调用的是 `EntityValidator.validateOrThrow(this.entityName, entityData)`，没有附带 `ValidationOptions`。
这意味着：
- 默认保存路径下，`groups` 过滤不会生效，因为 `EntityValidator` 收到的是空分组，等价于“所有规则都跑”
- `Unique()` 装饰器本身不会通过 `uniqueChecker` 执行唯一性判断；默认保存依赖写入层的额外唯一性校验，而不是 `EntityValidator` 的 `unique` 分支

### 4.3 非保存路径不会自动做字段验证
- 查询加载：`findById/findAll/findPaginated` 这类仓储查询会执行 `afterLoad` / `afterLoadBatch`，不走 `EntityValidator`
- `rawQuery/rawQuerySafe` 只做结果映射，不执行 `afterLoad` / `afterLoadBatch`
- 删除：执行 `beforeDelete`、删除动作、`afterDelete`，不走 `EntityValidator`
- 恢复：执行 `beforeRestore`、恢复动作；只有 `affectedRows > 0` 时才继续执行 `afterRestore`，不走 `EntityValidator`

### 4.4 批量插入不是单条保存的等价展开
批量插入路径与单条保存不同：
- 打开 `executeValidation` 时，会逐条执行 `EntityValidator.validateOrThrow`
- 打开 `executeHooks` 时，只会逐条执行 `HooksProcessor.executeBeforeSave`
- 不会自动执行 `beforeInsert`、`afterInsert`、`afterSave`

```text
batchInsert(entities, options)
  -> options.executeValidation ? 对每条数据执行 validateOrThrow : 跳过
  -> options.executeHooks ? 对每条数据执行 executeBeforeSave : 跳过
  -> batch insert
  -> 结束（没有 afterInsert / afterSave）
```

这不是文档口误，而是当前源码的真实行为。

## 5. 关键流程/规则
### 5.1 正例：手动编排一个与单条保存一致的顺序
```ts
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

EntityValidator.validateOrThrow('User', entityData)
await hooksProcessor.executeBeforeSave('User', entityData)
await hooksProcessor.executeBeforeInsert('User', entityData)
// insert
await hooksProcessor.executeAfterInsert('User', entityData)
await hooksProcessor.executeAfterSave('User', entityData)
```

### 5.2 正例：查询完成后只执行 `afterLoad`
```ts
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 数据已经从数据库映射完成 */

await hooksProcessor.executeAfterLoad('User', entityData)
```

### 5.3 正例：批量加载统一跑 `afterLoadBatch`
```ts
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const rows = [
  /* 多个实体数据对象 */
]

await hooksProcessor.executeAfterLoadBatch('User', rows)
```

### 5.4 误用示例：指望 `beforeSave` 先修数据，再让验证通过
```ts
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* email 仍然非法 */

// 误用：这种顺序不是默认保存链路
await hooksProcessor.executeBeforeSave('User', entityData)
EntityValidator.validateOrThrow('User', entityData)
```

真实保存路径是先 `validateOrThrow`，后 `executeBeforeSave`。如果字段先天不合法，你根本进不到 `beforeSave`。

### 5.5 误用示例：把批量插入当成“会自动跑完整生命周期”的保存
```ts
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

await hooksProcessor.executeBeforeSave('User', entityData)
// batch insert
await hooksProcessor.executeAfterSave('User', entityData)
```

这段写法只是手动补流程，不代表当前批量插入实现会自动这样做。当前源码默认只会在可选开关打开时执行 `beforeSave`，不会自动补 `afterSave`。

### 5.6 误用示例：以为删除和恢复也会自动校验字段规则
```ts
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

// 误判：删除链路实际只关心 beforeDelete / afterDelete
EntityValidator.validateOrThrow('User', entityData)
await hooksProcessor.executeBeforeDelete('User', entityData)
```

删除、恢复、查询不是字段入库校验场景。默认源码路径不会在这些阶段自动调用 `EntityValidator`。

## 6. 示例
### 6.1 通过错误类型判断当前失败发生在验证阶段还是钩子阶段
```ts
import { ValidationError } from 'ocorm'
import { HookExecutionError } from 'ocorm'
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

try {
  EntityValidator.validateOrThrow('User', entityData)
  await hooksProcessor.executeBeforeSave('User', entityData)
} catch (error) {
  if (error instanceof ValidationError) {
    console.error('失败发生在验证阶段:', error.message)
  } else if (error instanceof HookExecutionError) {
    console.error('失败发生在钩子阶段:', error.message)
  }
}
```

### 6.2 手动使用分组验证，再进入钩子
```ts
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

EntityValidator.validateOrThrow('User', entityData, {
  groups: ['create']
})
await hooksProcessor.executeBeforeSave('User', entityData)
```

这能做到“只跑 create 分组再进钩子”，但这是调用方手动编排的行为，不是默认保存链路。

### 6.3 手动提供 `uniqueChecker` 后再进入钩子
```ts
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

EntityValidator.validateOrThrow('User', entityData, {
  uniqueChecker: (_entityName, _propertyName, value) => value !== 'used@example.com'
})
await hooksProcessor.executeBeforeSave('User', entityData)
```

## 7. 验收与测试
最小验收应覆盖这些断言：
- 保存场景里，验证失败时 `beforeSave` 不会执行
- 保存场景里，`beforeSave` 抛错后，`beforeInsert` / `beforeUpdate` 不会继续执行
- 查询场景里，只触发 `afterLoad` / `afterLoadBatch`，不触发验证
- 删除与恢复场景里，只触发对应钩子，不触发验证
- 批量插入开启 `executeHooks` 时，只执行 `beforeSave`
- 如果业务需要 `groups` 或 `uniqueChecker`，必须由调用方手动传给 `EntityValidator`

```ts
import { ValidationError } from 'ocorm'
import { HookExecutionError } from 'ocorm'
import { EntityValidator } from 'ocorm'
import { HooksProcessor } from 'ocorm'

const hooksProcessor = HooksProcessor.getInstance()
const entityData = /* 已构造的实体数据对象 */

try {
  EntityValidator.validateOrThrow('User', entityData)
  await hooksProcessor.executeBeforeSave('User', entityData)
  await hooksProcessor.executeBeforeUpdate('User', entityData)
} catch (error) {
  if (error instanceof ValidationError) {
    console.error('先修规则，再看钩子')
  } else if (error instanceof HookExecutionError) {
    console.error('验证已通过，失败发生在生命周期阶段')
  }
}
```

## 8. 常见问题
### 8.1 为什么 `beforeSave` 没有执行，但我已经注册了钩子？
先看是不是 `EntityValidator.validateOrThrow` 已经抛了 `ValidationError`。默认保存链路里，验证失败会直接中断，钩子根本不会开始。

### 8.2 为什么我配置了 `groups`，但默认保存像是全部规则都在跑？
因为默认保存并没有把 `groups` 传给 `EntityValidator`。在 `EntityValidator` 视角里，这等价于空分组，结果就是所有规则执行。

### 8.3 为什么我给字段加了 `Unique()`，但默认保存没有走我自定义的唯一性检查？
因为默认保存不会把 `uniqueChecker` 传给 `EntityValidator`。如果你依赖装饰器级 `unique` 分支，必须手动调用 `EntityValidator.validateOrThrow(..., { uniqueChecker })`。

### 8.4 为什么批量插入后 `afterSave` 没有触发？
因为当前批量插入实现本来就不自动执行 `afterInsert` / `afterSave`。如果业务强依赖这些后置动作，要么改用逐条 `save`，要么在业务层显式补调。

### 8.5 报错与定位方式
```ts
import { ValidationError } from 'ocorm'
import { HookExecutionError } from 'ocorm'

function locateStage(error: object): string {
  if (error instanceof ValidationError) {
    return 'validation'
  }
  if (error instanceof HookExecutionError) {
    return 'hook'
  }
  return 'unknown'
}
```

定位顺序建议固定如下：
1. 先分辨错误类型，`ValidationError` 说明连 `before*` 都还没开始，`HookExecutionError` 说明验证已通过。
2. 如果是 `HookExecutionError`，再看消息里是 `beforeSave`、`beforeInsert`、`afterLoad` 还是别的阶段。
3. 如果问题发生在批量插入，额外确认是否只开了 `executeHooks`，以及你预期的是否其实是 `afterSave`。
4. 如果问题发生在查询、删除、恢复，默认应优先排查钩子，不要把字段验证当成入口。
