# 03-只读事务语义与回退策略

> 状态：已完成
> 适用版本：ocorm 3.x（按当前仓库实现）
> 最后更新：2026-03-28

## 1. 目标
- 说明 `TransactionOptions.readOnly()` 在当前仓库里的真实语义。
- 说明只读事务是如何通过 `PRAGMA query_only` 启用和恢复的。
- 明确“平台不支持只读事务”时当前实现的处理方式，不做空泛承诺。

## 2. 真实行为边界
当前只读事务不是逻辑标记，而是 `TransactionManager` 在 `beginTransaction()` 之后尝试执行：

```text
PRAGMA query_only = 1
```

事务结束后，如果只读模式曾成功启用，会在 `resetTransactionOptions(...)` 中恢复：

```text
PRAGMA query_only = 0
```

关键点只有三条：
- 启用时机：`BEGIN` 之后，不是之前
- 恢复时机：`COMMIT/ROLLBACK` 之后的事务选项重置阶段
- 失败策略：平台不支持 `query_only` 时直接抛 `TransactionRollbackError`

## 3. 正确示例
最安全的用法是直接通过 `TransactionOptions.readOnly()` 创建选项。

```arkts
import { Repository, TransactionOptions } from 'ocorm'

const repo = new Repository('User')

await repo.transactionWithOptions(async (txRepo) => {
  const rows = await txRepo.findAll()
  console.info(rows.length)
}, TransactionOptions.readOnly())
```

如果你要观测调用顺序，当前仓库已经提供测试观察点。

```arkts
import { TransactionManager, TransactionOptions } from 'ocorm'

const order: string[] = []
TransactionManager.resetTestHooks()
TransactionManager.setTransactionPhaseObserverForTest((phase: string) => order.push(phase))
TransactionManager.setReadOnlyModeObserverForTest((readOnly: boolean) => {
  order.push(readOnly ? 'Q1' : 'Q0')
})

await manager.transactionWithOptions(async () => {}, TransactionOptions.readOnly())
console.info(order)
```

从 `EntryOrmReviewFixSuite` 的断言口径看，正确顺序应理解为：

```text
BEGIN
Q1
... action ...
COMMIT / ROLLBACK
Q0
```

## 4. 误用示例
只读事务里执行写入，不会因为你“自觉只读”而被放过。底层 `query_only` 会直接让写入失败。

```arkts
import { Repository, TransactionOptions } from 'ocorm'

const repo = new Repository('User')

await repo.transactionWithOptions(async (txRepo) => {
  await txRepo.rawExecute('UPDATE user SET name = ? WHERE id = ?', ['blocked', 1])
}, TransactionOptions.readOnly())
```

另一个误用是把“只读不支持”当成“自动降级成功”。当前实现不会静默降级。

```arkts
import { TransactionOptions } from 'ocorm'

try {
  await manager.transactionWithOptions(async () => {}, TransactionOptions.readOnly())
} catch (error) {
  console.error('平台不支持 query_only 时会直接抛错，而不是自动成功', error)
}
```

## 5. 回退与失败语义
只读事务失败时，最关键的分支有两类：
- 平台不支持 `PRAGMA query_only`
- 已开启只读模式，但事务体或提交/回滚阶段失败

```arkts
import { TransactionOptions } from 'ocorm'

try {
  await manager.transactionWithOptions(async () => {
    throw new Error('mock failure')
  }, TransactionOptions.readOnly())
} catch (error) {
  console.error(error)
}
```

当前仓库里的真实语义是：
- 不支持 `query_only`：抛 `TransactionRollbackError('当前平台不支持只读事务（PRAGMA query_only）')`
- 事务体失败：先走回滚，再重置只读选项，最后对外抛回滚错误
- 恢复 `Q0` 失败：`resetTransactionOptions(...)` 只记录错误日志，不在该恢复阶段继续抛出

## 6. 与隔离级别的关系
只读事务和隔离级别是两套独立选项，但它们都通过 `TransactionManager` 在事务内应用。当前实现中：
- `READ_COMMITTED`：平台不支持 `read_uncommitted` 时按默认语义继续
- `READ_UNCOMMITTED`：平台不支持时显式失败
- `readOnly`：平台不支持 `query_only` 时显式失败

```arkts
import { IsolationLevel, TransactionOptions } from 'ocorm'

const options = TransactionOptions.fromConfig({
  isolation: IsolationLevel.READ_COMMITTED,
  readOnly: true,
  timeout: 3000
})
```

## 7. 排障建议
- 如果报错里包含 `query_only` / `readonly` / `只读事务`，先看平台是否支持 `PRAGMA query_only`
- 如果用例断言顺序不对，优先对照 `EntryOrmReviewFixSuite` 中的 `BEGIN -> Q1 -> Q0`
- 只读事务里不要做任何 `rawExecute/save/remove/attach/sync`
- 不要把“只读”当成权限系统，它只是当前连接事务级的 SQL 写保护
