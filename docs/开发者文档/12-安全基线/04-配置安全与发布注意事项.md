# 04-配置安全与发布注意事项

> 状态：已完成
> 适用范围：构建配置、模块声明、日志与发布
> 最后更新：2026-03-28

## 1. 目标
- 把当前仓库里已经存在的配置安全点讲清楚。
- 区分“本地可用配置”和“适合提交入库的配置”。
- 说明发布前需要核对的签名、混淆、权限与日志开关。

## 2. 当前仓库中的安全敏感配置
本仓库最敏感的配置不在代码逻辑，而在构建与签名文件：
- 根目录 `build-profile.json5`
- `OCORM/build-profile.json5`
- `entry/src/main/module.json5`
- `OCORM/src/main/module.json5`

根 `build-profile.json5` 当前直接包含本机签名材料路径和密码散列，这类内容只能作为本地开发配置，不能当作可移植模板。

```json5
{
  "app": {
    "signingConfigs": [
      {
        "name": "default",
        "material": {
          "certpath": "C:\\Users\\hskns\\.ohos\\config\\...cer",
          "profile": "C:\\Users\\hskns\\.ohos\\config\\...p7b",
          "storeFile": "C:\\Users\\hskns\\.ohos\\config\\...p12"
        }
      }
    ]
  }
}
```

库模块 `OCORM/build-profile.json5` 当前在 `release` 下启用了混淆规则，这属于应该保留的发布配置。

```json5
{
  "buildOptionSet": [
    {
      "name": "release",
      "arkOptions": {
        "obfuscation": {
          "ruleOptions": {
            "enable": true,
            "files": ["./obfuscation-rules.txt"]
          },
          "consumerFiles": ["./consumer-rules.txt"]
        }
      }
    }
  ]
}
```

## 3. 正确做法
业务和库模块的发布配置应至少满足：
- 本地签名材料不外泄
- `release` 开启混淆
- 模块类型和导出范围符合预期
- 运行时日志不在发布模式下无限制开启

```arkts
import { DatabaseConfig, LogLevel } from 'ocorm'

const config = new DatabaseConfig('app.db')
config.enableLogger = false
config.loggerLevel = LogLevel.ERROR
```

Entry 模块当前公开的是 `EntryAbility`，`EntryBackupAbility` 则是 `exported: false`，这是合理的最小暴露面。

```json5
{
  "module": {
    "abilities": [
      {
        "name": "EntryAbility",
        "exported": true
      }
    ],
    "extensionAbilities": [
      {
        "name": "EntryBackupAbility",
        "exported": false
      }
    ]
  }
}
```

## 4. 误用示例
最典型的误用是把本机签名配置直接长期保存在可共享配置中。

```json5
{
  "material": {
    "keyPassword": "0000001A....",
    "storePassword": "0000001A...."
  }
}
```

另一个误用是发布版继续开启详细日志，把原始错误或 SQL 直接打出去。

```arkts
const config = new DatabaseConfig('prod.db')
config.enableLogger = true
config.loggerLevel = LogLevel.DEBUG
```

还有一种误用是为了“省事”把不该开放的能力声明成 `exported: true`。

```json5
{
  "extensionAbilities": [
    {
      "name": "EntryBackupAbility",
      "exported": true
    }
  ]
}
```

## 5. 发布前核对清单
- 根 `build-profile.json5` 中的签名材料是否仍是本机开发路径
- `OCORM/build-profile.json5` 的 `release` 混淆是否开启
- `entry/src/main/module.json5` 是否只暴露必要入口
- `DatabaseConfig`、`Logger` 是否在发布环境关闭不必要的 debug 输出
- `OHPM` 依赖版本是否与当前 `README/CHANGELOG` 口径一致

```bash
rg -n "signingConfigs|storePassword|keyPassword|exported|obfuscation|enableLogger|loggerLevel" build-profile.json5 OCORM/build-profile.json5 entry/src/main/module.json5 OCORM/src/main/module.json5 OCORM/src/main/ets
```

## 6. 安全建议
- 把签名材料路径与密码配置视为环境私有信息，不要拿当前根 `build-profile.json5` 当共享模板
- 生产发布时优先保留 `release` 混淆，并结合 `consumer-rules.txt` 维护对外可用面
- 日志中涉及用户标识、SQL 或连接细节时，优先配合 `ErrorSanitizer` 和受控日志级别
- 模块导出面遵循最小暴露，`entry` 负责入口，`ocorm` 作为 `har` 不额外声明能力
