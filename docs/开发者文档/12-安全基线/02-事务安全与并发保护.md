# 02-事务安全与并发保护

> 状态：已落地
> 适用版本：ocorm 3.x（按当前仓库源码）
> 最后更新：2026-03-28

## 1. 目标

本基线只描述当前仓库已经实现的事务安全与并发保护，不引入未落地的抽象承诺。

- 同一连接上的事务必须串行执行，避免交叉提交和回滚污染
- Repository 在活动事务存在时必须阻断未绑定事务上下文的写入与嵌套事务启动
- 超时、回滚、重试和后台清理的行为必须按源码现状理解，不能靠猜

## 3. 实现级安全规则

| 主题 | 已实现行为 | 安全含义 |
| --- | --- | --- |
| 同连接并发控制 | `TransactionManager` 通过 `WeakMap<RdbStore, ConnectionTransactionLock>` 串行化同一 `store` 上的事务 | 同一连接不会并发进入两个事务回调 |
| 活动事务跟踪 | `activeTransactions` 跟踪每个连接的活动事务计数，`activeTransactionCount` 与 `pendingCleanupCount` 提供全局视角 | 可以识别“连接级事务仍在进行”或“超时后清理仍未完成” |
| 写路径保护 | `RepositoryTransactionGuardSupport.ensureWritePathAllowed()` 在三种情况下拒绝写入：未绑定但处于严格事务中、连接池模式下存在活动事务、当前连接已激活事务 | 防止绕开事务上下文直接写库 |
| 嵌套事务保护 | `ensureNoNestedTransaction()` 会阻断同一 Repository 嵌套调用、连接池活动事务下再开事务、跨 Repository 嵌套事务 | 当前公开 API 不支持嵌套事务；写路径内部在需要原子回滚时会使用 `SAVEPOINT` |
| 超时清理 | `transactionWithOptions()` 超时后不会立即释放锁；它会把清理留到后台，等清理 Promise 结束后再释放锁和 lease | 防止超时事务在后台继续执行时被其他事务抢占同一连接 |
| 重试边界 | 仅普通执行失败允许按 `retries` 重试；配置类错误和超时不会继续重试 | 减少把错误配置放大成重复失败 |
| 隔离级别边界 | 仅支持 `READ_UNCOMMITTED` 与 `READ_COMMITTED`；`REPEATABLE_READ` / `SERIALIZABLE` 不提供强语义 | 调用方不得假设更强隔离已实现 |

需要明确的一点：当前锁是“连接级串行”，不是“全局串行”。如果连接池发放的是不同 `RdbStore`，事务可以并发；如果要阻断未绑定写入，必须结合连接池保护策略和 Repository 守卫一起用。

## 4. 正确示例

### 4.1 连接池与未绑定写入保护配置

```ts
import { DatabaseConfig } from '../../src/main/ets/database/DatabaseConfig'
import { LogLevel } from '../../src/main/ets/logging/LogLevel'

const config = new DatabaseConfig('app.db')
  .setConnectionPool({
    enabled: true,
    minSize: 1,
    maxSize: 4,
    acquireTimeoutMs: 3000,
    idleTimeoutMs: 10000,
    healthCheckIntervalMs: 0
  })
  .setPooledWriteGuardMode('global')

config.enableLogger = true
config.loggerLevel = LogLevel.INFO
```

这段配置只做两件真实存在的事：启用连接池，并把 `pooledWriteGuardMode` 切到 `global`。源码没有提供更多模式，只有 `'global' | 'off'`。

### 4.2 带超时与重试的事务执行

```ts
import { TransactionManager } from '../../src/main/ets/repository/TransactionManager'
import { IsolationLevel, TransactionOptions } from '../../src/main/ets/repository/TransactionOptions'
import { Logger } from '../../src/main/ets/logging/Logger'
import { LogLevel } from '../../src/main/ets/logging/LogLevel'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.INFO)

const txManager = new TransactionManager(logger, () => store)
const options = TransactionOptions.fromConfig({
  isolation: IsolationLevel.READ_COMMITTED,
  timeout: 5000,
  retries: 1,
  retryDelay: 200,
  readOnly: false
})

await txManager.transactionWithOptions(async (_txStore) => {
  // 这里必须复用事务回调提供的 store / Repository 执行业务写入
  // 不要在回调外另拿一个 Repository 直接写库
}, options)
```

### 4.3 Repository 写路径和嵌套事务守卫

```ts
import { RepositoryTransactionGuardSupport } from '../../src/main/ets/repository/RepositoryTransactionGuardSupport'
import { TransactionManager } from '../../src/main/ets/repository/TransactionManager'

const guard = new RepositoryTransactionGuardSupport(
  () => false,
  () => TransactionManager.hasAnyActiveTransaction(),
  () => store
)

guard.ensureNoNestedTransaction('transaction')
guard.ensureWritePathAllowed('save')

// 只有在没有严格事务、没有连接池活动事务、且当前连接未激活事务时，这两个检查才会放行
```

### 4.4 事务阶段观测与串行行为验证

```ts
import { TransactionManager } from '../../src/main/ets/repository/TransactionManager'

const phases: string[] = []
TransactionManager.resetTestHooks()
TransactionManager.setTransactionPhaseObserverForTest((phase: string) => {
  phases.push(phase)
})

await txManager.transaction(async () => {
  // 执行业务逻辑
})

// phases 里至少会出现 BEGIN -> COMMIT，异常路径则会出现 BEGIN -> ROLLBACK
```

## 5. 误用示例

### 5.1 误以为库支持更强隔离级别

```ts
import { TransactionOptions } from '../../src/main/ets/repository/TransactionOptions'

const options = TransactionOptions.serializable()

// 这里不会返回事务选项，而是立即抛出 Error：
// 当前不支持 SERIALIZABLE 隔离级别，请使用 READ_COMMITTED 或 READ_UNCOMMITTED
```

### 5.2 在活动事务存在时，绕开事务上下文直接写入

```ts
import { RepositoryTransactionGuardSupport } from '../../src/main/ets/repository/RepositoryTransactionGuardSupport'

const guard = new RepositoryTransactionGuardSupport(
  () => false,
  () => true,
  () => store
)

guard.ensureWritePathAllowed('update')

// 会抛 TransactionRollbackError：
// 检测到连接池模式下存在活动事务，禁止在未绑定事务上下文的 Repository 执行 update 写入，请复用事务回调提供的 Repository 实例
```

### 5.3 在同一 Repository 或跨 Repository 再开事务

```ts
const nestedGuard = new RepositoryTransactionGuardSupport(
  () => true,
  () => false,
  () => store
)

nestedGuard.ensureNoNestedTransaction('transactionWithOptions')

// 会抛 TransactionRollbackError：
// 不支持同一 Repository 嵌套调用 transactionWithOptions，请复用外层事务或改为 SAVEPOINT 语义
```

### 5.4 误以为超时后还能继续重试

```ts
const options = TransactionOptions.withTimeout(100)
options.retries = 3

await txManager.transactionWithOptions(async () => {
  await new Promise<void>((resolve) => setTimeout(resolve, 500))
}, options)

// 当前实现里，超时会直接转成后台清理并抛错，事务不会继续下一轮重试
```

## 6. 失败语义与排障

### 6.1 真实失败路径

```text
获取连接失败
-> logger.logError("获取数据库连接失败: ...")
-> 抛 TransactionRollbackError("获取数据库连接失败: ...")

事务回调抛错
-> 若事务已开始则先 rollBack()
-> 回滚成功：logger.logTransaction('ROLLBACK')
-> 回滚失败：logger.logError("事务回滚失败: ...")
-> 抛 TransactionRollbackError("原始错误: ..., 回滚错误: ...")

事务超时
-> logger.logError("事务超时，转入后台清理: ...ms")
-> pendingCleanupCount + 1
-> 后台清理结束后释放 lock 和 lease
-> 对外抛 TransactionRollbackError("事务超时 (...ms)")
```

### 6.2 配置错误不会被重试

`TransactionManager.isNonRetryableConfigurationError()` 已经把下面几类错误列为不可重试：

- `不支持的事务隔离级别`
- `当前平台不支持 READ_UNCOMMITTED 隔离级别`
- `当前平台不支持只读事务`
- `设置事务隔离级别失败`
- `设置事务只读模式失败`

也就是说，重试机制只适用于普通执行失败，不适用于能力缺失或事务配置错误。

### 6.3 失败日志与本地化排障示例

```ts
import { ERROR_TRANSACTION_ROLLBACK } from '../../src/main/ets/errors/ErrorCodes'
import { ErrorTemplateContext, formatErrorMessage, setErrorLocale } from '../../src/main/ets/errors/ErrorLocale'
import { sanitizeErrorMessage } from '../../src/main/ets/errors/ErrorSanitizer'
import { Logger } from '../../src/main/ets/logging/Logger'
import { LogLevel } from '../../src/main/ets/logging/LogLevel'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.ERROR)
setErrorLocale('zh')

try {
  await txManager.transactionWithOptions(async () => {
    throw new Error('patient_id=12345678, nested transaction denied')
  }, TransactionOptions.createDefault())
} catch (error) {
  const context = new ErrorTemplateContext()
  context.details = sanitizeErrorMessage(error instanceof Error ? error.message : String(error))
  const localized = formatErrorMessage(
    ERROR_TRANSACTION_ROLLBACK,
    '事务已回滚: {details}',
    context
  )
  logger.logError(localized)
}
```

### 6.4 排障顺序

1. 先区分是连接获取失败、事务体失败、回滚失败还是超时进入后台清理
2. 再确认是否误用了不支持的隔离级别，特别是 `SERIALIZABLE` 和 `REPEATABLE_READ`
3. 如果是“活动事务存在时写入被拒绝”，检查是否复用了事务回调里的 Repository/Store
4. 如果是超时，先用 `TransactionManager.hasAnyActiveTransaction()` 判断是否还有后台清理未完成
5. 线上排障只看脱敏后的错误详情，不要把原始 SQL 或原始标识直接打印出来

## 7. 验收清单

- 同一连接上的事务入口统一经 `TransactionManager` 调度，没有私自 `beginTransaction()` / `commit()` / `rollBack()` 分叉实现
- `SERIALIZABLE` / `REPEATABLE_READ` 没有被当成已实现能力写进业务假设
- Repository 写入与开启事务前都经过事务守卫，避免事务边界外写入与嵌套事务
- 事务超时时，调用方能识别“对外已失败，但后台仍可能在清理”的状态
- 事务失败日志统一经过脱敏后再输出
