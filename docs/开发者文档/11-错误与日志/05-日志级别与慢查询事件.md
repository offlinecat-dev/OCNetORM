# 05-日志级别与慢查询事件

> 状态：已完成
> 适用版本：ocorm 3.x（当前 3.0.2）
> 最后更新：2026-03-27

## 1. 目标
- 说明 `logging/LogLevel.ets` 与 `logging/Logger.ets` 的真实日志级别语义。
- 说明查询日志、事务日志、错误日志、告警日志、调试日志的门控规则与格式。
- 说明“慢查询事件”在当前公开 API 范围内的真实状态：没有独立事件类型，只有带耗时的 `logQuery(sql, duration)` 记录，慢查询需要由调用方按耗时阈值自行判定。

## 2. 级别模型
`LogLevel.ets` 只定义了 3 个等级：
- `LogLevel.ERROR = 0`
- `LogLevel.INFO = 1`
- `LogLevel.DEBUG = 2`

`Logger.ets` 内部通过 `shouldLog(requiredLevel)` 做门控，判断条件只有一条：`enabled && currentLevel >= requiredLevel`。因此日志能力是单向继承的：
- `DEBUG` 会放行 `ERROR`、`INFO`、`DEBUG` 级别对应的方法。
- `INFO` 会放行 `ERROR` 和 `INFO`。
- `ERROR` 只会放行 `ERROR`。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)

console.info(logger.isEnabled())
console.info(logger.getLevel() === LogLevel.DEBUG)
```

## 3. 各日志方法的门控规则
当前 `Logger` 对外暴露的方法可以按门控分成三组。

只在 `DEBUG` 级别记录：
- `logQuery(sql, duration)`
- `logDebug(debugMessage)`
- `logTaskPool(taskName, duration, dataCount, usedTaskPool)`
- `logPagination(page, pageSize, total, duration)`
- `logParallelRelation(relationNames, duration, successCount)`
- `logBatchInsert(tableName, totalCount, insertedCount, duration, useTransaction)`

在 `INFO` 及以上记录：
- `logTransaction(action)`
- `logWarn(warnMessage)`
- `logInfo(infoMessage)`

只要日志启用就会记录：
- `logError(errorMessage)`

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.INFO)
logger.setRecordHistory(true)
logger.clearLogHistory()

logger.logError('error')
logger.logTransaction('BEGIN')
logger.logInfo('service booted')
logger.logDebug('debug detail')
logger.logQuery('SELECT 1', 1)

const history = logger.getLogHistory()
console.info(history.length) // 预期只包含 ERROR/TRANSACTION/INFO
```

## 4. 日志格式与历史缓冲
`Logger.formatMessage(...)` 的输出格式固定为：
`[ocorm][类型][耗时] 消息`

关键点：
- 查询日志的耗时字段是 `${duration}ms`。
- 非耗时日志一般传 `-`。
- `setRecordHistory(true)` 后，`Logger` 会把最近日志写入内存历史。
- 历史上限是 200 条，超过后采用先进先出。

这也是当前最可靠的本地观测手段，因为用户通常看不到 `hilog` 的即时输出。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)
logger.setRecordHistory(true)
logger.clearLogHistory()

logger.logDebug('bootstrap')
logger.logQuery('SELECT * FROM users', 15)

const history = logger.getLogHistory()
for (let i = 0; i < history.length; i++) {
  console.info(history[i])
}
```

## 5. 慢查询事件的真实语义
这里必须说清楚，避免文档继续扩散幻觉。

在当前允许使用的真实 API 范围内：
- 没有 `SlowQueryEvent` 类型。
- 没有 `logSlowQuery(...)` 方法。
- 没有 `setSlowQueryThreshold(...)`、`onSlowQuery(...)` 这类监听接口。

当前所谓“慢查询事件”，本质上只能由调用方基于 `logQuery(sql, duration)` 的 `duration` 参数自行判定。也就是说，`Logger` 只负责记录查询和耗时，不负责替你定义“多慢算慢”。

工程上的正确理解是：
- `logQuery` 是查询执行事件。
- “慢查询”是你在业务或测试层根据阈值对该事件做二次分类后的结果。
- 如果当前运行级别不是 `DEBUG`，查询日志不会留下，自然也无法基于 `Logger` 观测慢查询。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)

function logSlowAwareQuery(sql: string, duration: number, thresholdMs: number): void {
  logger.logQuery(sql, duration)
  if (duration >= thresholdMs) {
    logger.logWarn(`slow query detected: ${duration}ms >= ${thresholdMs}ms`)
  }
}

logSlowAwareQuery("SELECT * FROM visits WHERE phone = '13800138000'", 1280, 1000)
```

## 6. 正确处理方式
围绕级别与慢查询，正确做法如下：
- 生产环境默认用 `INFO` 或 `ERROR`，不要长期保持 `DEBUG`。
- 需要复现查询性能问题时，短时切到 `DEBUG`，并用 `setRecordHistory(true)` 收集日志。
- 慢查询告警由调用方基于 `duration` 自己判定，判定后用 `logWarn()` 或 `logError()` 输出摘要。
- 需要排障时优先保留 `QUERY + WARN/ERROR` 两条链路：前者给耗时，后者给结论。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)
logger.setRecordHistory(true)

function recordQuery(sql: string, duration: number): void {
  logger.logQuery(sql, duration)
  if (duration >= 500) {
    logger.logWarn(`slow query summary: ${duration}ms`)
  }
}

recordQuery('SELECT * FROM orders', 42)
recordQuery('SELECT * FROM orders WHERE status = 1', 860)
```

## 7. 误处理示例
下面这些写法都不成立：
- 在 `INFO` 级别下期待 `logQuery()` 自动留下慢查询轨迹。
- 以为 `Logger` 自带慢查询阈值配置，实际上源码没有。
- 在生产环境长期开启 `DEBUG`，把所有 SQL 明细都打出来。
- 只打 `logWarn('slow query')`，却不保留对应的查询耗时和上下文。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.INFO)

// 反例 1：INFO 下 logQuery 根本不会记录
logger.logQuery('SELECT * FROM users', 1200)

// 反例 2：假设 Logger 内部会自动判定慢查询
logger.logInfo('waiting for slow query event')

// 反例 3：没有查询明细，只剩一句空泛告警
logger.logWarn('slow query happened')
```

## 8. 观测与排障方式
最实用的排障方式是直接检查历史缓冲区：
1. `configure(true, LogLevel.DEBUG)`，否则看不到 `QUERY`。
2. `setRecordHistory(true)` 并在复现场景前 `clearLogHistory()`。
3. 执行操作后读取 `getLogHistory()`。
4. 搜索 `[QUERY]`、`[WARN]`、`[ERROR]`，按时间顺序比对。
5. 如果需要验证格式，可以检查是否含有 `[ocorm]` 前缀、日志类型和 `${duration}ms`。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

const logger = Logger.getInstance()
logger.configure(true, LogLevel.DEBUG)
logger.setRecordHistory(true)
logger.clearLogHistory()

logger.logQuery('SELECT * FROM users WHERE id = 1', 8)
logger.logQuery('SELECT * FROM users WHERE id = 2', 1100)
logger.logWarn('slow query summary: 1100ms')

const history = logger.getLogHistory()
for (let i = 0; i < history.length; i++) {
  console.info(history[i])
}
```

## 9. 补充风险
- `logQuery()` 会自动对 SQL 中的字符串、敏感键值对和 4 位及以上数字做脱敏，但这不等于可以无脑打开 `DEBUG`。
- `logInfo()` 和 `logDebug()` 不会自动做敏感脱敏，性能诊断摘要里不要塞原始秘密。
- 如果你的慢查询判定完全依赖 `Logger` 历史，那么关闭 `recordHistory` 后本地测试就看不到留痕。
- `Logger.resetInstance()` 仅适用于测试隔离，不应作为运行时热切换方案。

```ts
import { Logger } from 'ocorm'
import { LogLevel } from 'ocorm'

Logger.resetInstance()
const logger = Logger.getInstance()
logger.configure(true, LogLevel.ERROR)
logger.setRecordHistory(true)
logger.logError('only error survives at ERROR level')
console.info(logger.getLogHistory().length)
```

## 10. 代码评审检查项
- 是否把日志级别设置与环境目的对齐，而不是全环境常驻 `DEBUG`。
- 是否明确知道 `logQuery()` 只在 `DEBUG` 级别生效。
- 是否把慢查询识别写在调用方，而不是假定 `Logger` 内部自带阈值系统。
- 是否在观测慢查询时同时保留查询日志和摘要告警。
