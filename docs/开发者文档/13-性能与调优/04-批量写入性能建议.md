# 04-批量写入性能建议

> 状态：已完成
> 适用范围：`batchInsert`、`saveAll`、事务与验证/钩子开关
> 最后更新：2026-03-28

## 1. 目标
- 说明当前仓库下批量写入该怎么选入口。
- 说明 `BatchInsertOptions` 的性能含义。
- 避免把“快”建立在错误语义上。

## 2. 入口选择
- 需要吞吐：优先 `batchInsert`
- 需要逐条复用 `save` 语义：用 `saveAll`
- 需要最强完整性：`BatchInsertOptions.createSafe()`
- 需要快速导入：`BatchInsertOptions.createFast()`

```arkts
import { BatchInsertOptions } from 'ocorm'

const fast = BatchInsertOptions.createFast()
const safe = BatchInsertOptions.createSafe()
console.info(fast, safe)
```

## 3. 正确示例
高吞吐导入先用 `batchInsert`，不要循环 `save`。

```arkts
const result = await repo.batchInsert(entities, BatchInsertOptions.createDefault())
console.info(result.insertedCount, result.totalCount)
```

如果你明确接受跳过事务、钩子、验证的风险，才使用 `createFast()`。

```arkts
const result = await repo.batchInsert(entities, BatchInsertOptions.createFast())
```

## 4. 误用示例
把 `saveAll` 当“高性能批量导入”是最常见误用。

```arkts
await repo.saveAll(entities)
```

另一个误用是把 `createFast()` 当成“只是少一点事务开销”。当前实现里它会同时关闭事务、钩子、验证。

```arkts
const options = BatchInsertOptions.createFast()
console.info(options.useTransaction, options.executeHooks, options.executeValidation)
```

## 5. 观察指标
- `insertedCount / totalCount`
- `failedIndexes`
- 单批总耗时
- 是否命中唯一性校验额外查询

```text
性能与正确性一起看:
1. 吞吐高不代表语义正确
2. 失败索引不为空时，先排查数据与约束
3. 批量写入后缓存失效与后续查询命中率也要看
```
