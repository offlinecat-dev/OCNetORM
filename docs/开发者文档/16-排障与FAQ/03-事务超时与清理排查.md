# 03-事务超时与清理排查

> 状态：已完成
> 适用范围：事务超时、后台清理、锁释放时序
> 最后更新：2026-03-28

## 1. 典型现象
当前仓库中，事务超时最容易表现为：
- 调用方很快收到 `事务超时 (xxxms)` 或 `BEGIN 观察 超时`
- 后续短时间内又出现“仍有活动事务”或并发写入被阻断
- 测试里表现为 `TransactionWithOptions_ShouldRejectConcurrentCall`、`TransactionTimeout_ShouldRejectOverlapBeforeCleanup`

## 2. 当前实现怎么处理超时
`TransactionManager.transactionWithOptionsInternal(...)` 不是超时即取消，而是：

```text
Promise.race(executionPromise, timeoutPromise)
-> timeout 先返回
-> 记录 "事务超时，转入后台清理"
-> pendingCleanupCount + 1
-> executionPromise 在后台继续收尾
-> 清理结束后 pendingCleanupCount - 1
-> 最后才释放 lock 和 lease
```

这意味着调用方看到的“超时”不等于底层连接已经立即完全空闲。

## 3. 正确复现方式

```arkts
import { Repository, TransactionOptions } from 'ocorm'

const repo = new Repository('User')

await repo.transactionWithOptions(async (_txRepo) => {
  await new Promise<void>((resolve) => setTimeout(resolve, 200))
}, TransactionOptions.withTimeout(20))
```

如果需要观测超时后的活动事务状态，可以直接看当前实现暴露的统计入口。

```arkts
import { TransactionManager } from 'ocorm'

const active = TransactionManager.hasAnyActiveTransaction()
console.info(active)
```

测试里也可以挂相位观察点确认是否出现后台清理。

```arkts
import { TransactionManager } from 'ocorm'

const phases: string[] = []
TransactionManager.resetTestHooks()
TransactionManager.setTransactionPhaseObserverForTest((phase: string) => phases.push(phase))
```

## 4. 误用示例
超时后立刻再复用同一逻辑继续写入，是最容易踩中的误用。

```arkts
import { EntityData, EntityDataInput, Repository, TransactionOptions } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const repo = new Repository('User')

try {
  await repo.transactionWithOptions(async () => {
    await new Promise<void>((resolve) => setTimeout(resolve, 500))
  }, TransactionOptions.withTimeout(10))
} catch (_error) {
  // 误用：立刻假设连接已完全释放
  await repo.save(makeUser(1, 'too-early'))
}
```

另一个误用是把超时和重试叠加后，期待事务自动进入下一轮。

```arkts
const options = TransactionOptions.withTimeout(10)
options.retries = 2

await repo.transactionWithOptions(async () => {
  await new Promise<void>((resolve) => setTimeout(resolve, 300))
}, options)
```

当前实现里，超时分支直接走后台清理，不会继续下一轮重试。

## 5. 真实排查顺序
- 先看日志里有没有 `事务超时，转入后台清理`
- 再看 `TransactionManager.hasAnyActiveTransaction()` 是否仍为 `true`
- 若并发调用被阻断，确认是否是前一个超时事务仍在后台清理
- 若长时间不释放，检查事务体里是否有外部长耗时调用、迟到提交、未结束的底层 SQL

```text
建议排查顺序:
1. timeout 配置值是否过低
2. 事务体是否包含 sleep / 远程调用 / 大批量写入
3. 是否命中了 pendingCleanupCount > 0
4. 是否立刻发起了下一次 transaction / save / rawExecute
5. 是否有额外日志显示 rollback / cleanup 异常
```
