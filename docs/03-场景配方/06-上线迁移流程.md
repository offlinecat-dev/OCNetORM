# 06-上线迁移流程

> 状态：已完成  
> 适用版本：ocorm 3.x（当前仓库）
> 最后更新：2026-03-28

## 场景问题
上线时你要把数据库结构从旧版本升级到新版本，并保证：
1. 版本推进与实际结构一致
2. 失败能回滚且版本不误前移
3. 可以在故障后精确定位哪一步失败

## 完整方案
用 `MigrationManager` 统一注册和执行迁移，发布阶段走 `migrateToLatest(store)`，回滚阶段走 `migrateTo(store, targetVersion)`。

```arkts
import { relationalStore } from '@kit.ArkData'
import { Migration, MigrationExecutionResult, MigrationManager } from 'ocorm'

class V1CreateUsersTable implements Migration {
  version: number = 1
  description: string = '创建 users 表'

  async up(store: relationalStore.RdbStore): Promise<void> {
    await store.executeSql('CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY AUTOINCREMENT, user_name TEXT)')
  }

  async down(store: relationalStore.RdbStore): Promise<void> {
    await store.executeSql('DROP TABLE IF EXISTS users')
  }
}

class V2AddEmailColumn implements Migration {
  version: number = 2
  description: string = 'users 增加 user_email 列'

  async up(store: relationalStore.RdbStore): Promise<void> {
    await store.executeSql('ALTER TABLE users ADD COLUMN user_email TEXT')
  }

  async down(store: relationalStore.RdbStore): Promise<void> {
    // SQLite 无法直接 DROP COLUMN，使用重建表策略
    await store.executeSql('CREATE TABLE IF NOT EXISTS __tmp_users_down (id INTEGER PRIMARY KEY AUTOINCREMENT, user_name TEXT)')
    await store.executeSql('INSERT INTO __tmp_users_down (id, user_name) SELECT id, user_name FROM users')
    await store.executeSql('DROP TABLE users')
    await store.executeSql('ALTER TABLE __tmp_users_down RENAME TO users')
  }
}

export async function runReleaseMigrations(store: relationalStore.RdbStore): Promise<MigrationExecutionResult> {
  const manager = new MigrationManager({
    enableLog: true,
    logTableName: '_migrations'
  })

  manager.registerAll([
    new V1CreateUsersTable(),
    new V2AddEmailColumn()
  ])

  const result = await manager.migrateToLatest(store)
  if (!result.success) {
    throw new Error(
      `迁移失败: from=${result.fromVersion}, to=${result.toVersion}, actual=${result.actualVersion}, error=${result.errorMessage}`
    )
  }
  return result
}
```

## 验证
上线前后至少验证：
1. 第一次执行成功后，再跑一次 `executedCount === 0`
2. 失败时 `actualVersion` 不应前移到 `toVersion`
3. `_orm_version` 表存在并可读

```arkts
import { relationalStore } from '@kit.ArkData'
import { MigrationManager } from 'ocorm'

async function verifyMigrationIdempotent(store: relationalStore.RdbStore, manager: MigrationManager): Promise<void> {
  const first = await manager.migrateToLatest(store)
  const second = await manager.migrateToLatest(store)

  console.info(first.success, first.executedCount)
  console.info(second.success, second.executedCount) // 预期 0

  const currentVersion = await manager.getCurrentVersionFromDb(store)
  console.info(currentVersion)
}
```

## 失败修复
1. 失败：`register()` 重复版本直接抛错  
修复：批量注册用 `registerAll()`，或在注册前做版本去重。

2. 失败：迁移失败后把 `toVersion` 当实际版本  
修复：失败场景只认 `actualVersion`，并优先排查 `errorMessage` 与 `_migrations` 日志。

3. 失败：手工改 `_orm_version` 导致“版本号已升级但结构没升级”  
修复：禁止手工改版本表，统一走 `MigrationManager` 执行链路。

4. 失败：需要回滚却手工逐条执行 `down`  
修复：使用 `migrateTo(store, targetVersion)`，由框架按注册顺序倒序回滚。
