# 01-用户表最小闭环

> 状态：已完成
> 适用版本：ocorm 3.x（当前仓库）
> 最后更新：2026-03-27

## 1. 目标
说明 `Repository.ets` 中最常用的 CRUD 入口及其真实保存语义，重点覆盖：
- `save` 如何决定插入或更新
- `findById`、`findAll` 的默认查询边界
- `remove`、`removeById`、`restore` 的返回约定
- `saveAll` 与事务语义的真实关系

## 2. 背景与适用范围
`Repository` 是实体级读写入口。对于单条写入、按主键查询、批量顺序保存、软删除恢复，这一层已经封装了缓存、钩子、级联和事务守卫。

本页只讨论 `Repository.ets` 暴露出来的 CRUD 语义，不扩展 `QueryBuilder` 或底层 `RdbStore` 的自定义行为。

## 3. 核心 API 速记
- `new Repository(entityName: string)`
- `save(entityData)`
- `findById(id, includeDeleted = false, useCache = true)`
- `findAll(includeDeleted = false)`
- `remove(entityData)`
- `removeById(id)`
- `restore(id)`
- `saveAll(entities)`

## 4. 关键语义
### 4.1 `save` 的真实行为
- `save` 的注释明确定义为“根据主键值判断是 INSERT 还是 UPDATE”。
- 调用前会先经过写入守卫；在连接池存在活动事务、或当前连接已有外层事务而你却使用了未绑定的 `Repository` 写入时，会直接拒绝。
- `save` 返回结果对象；但如果当前仓库处在严格事务上下文中，也就是事务回调里传入的 `txRepo`，失败结果会被升级为异常抛出，而不是安静地返回 `success = false`。

### 4.2 查询默认值
- `findById` 默认 `includeDeleted = false`，如果实体启用了软删除，默认查不到已软删数据。
- `findById` 默认 `useCache = true`，只在“不包含已删除数据”的路径上读写缓存。
- `findAll` 同样默认排除软删除数据，并在结果返回前批量执行 `afterLoad` 钩子。

### 4.3 删除与恢复
- `remove(entityData)` 会先执行 `beforeDelete`、级联删除，再内部转到 `removeById(pk)`；成功后才执行 `afterDelete`。
- `removeById(id)` 直接按主键删除/软删，不会自动回补完整实体，也不会执行 `beforeDelete/afterDelete`。
- `restore(id)` 只用于恢复软删除记录；若实体未启用软删除，底层删除操作会抛错。
- `restore(id)` 会先执行 `beforeRestore`；只有 `affectedRows > 0` 时才继续执行 `afterRestore`。
- 在严格事务中，`remove` / `removeById` / `restore` 的失败结果同样会被升级为异常。

### 4.4 `saveAll` 不是原子批处理
- `saveAll` 位于 `BatchOperations.ets`，实现是逐条调用 `saveOne`，按顺序收集每一项结果。
- 这意味着它不是数据库级 `batchInsert`，也不是自动事务包裹的“全成或全败”。
- 如果你需要真正的批量插入性能或显式事务边界，应改用 `batchInsert` 或外层 `transaction`。

## 5. 正确示例
下面的用法符合 `Repository.ets` 当前实现：先保存，再按主键回读，最后按需删除或恢复。

```ts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const userRepo = new Repository('User')

const saveResult = await userRepo.save(makeUser(1, 'Alice'))
if (!saveResult.success) {
  console.error('save 失败:', saveResult.errorMessage)
}

const current = await userRepo.findById(1)
const allUsers = await userRepo.findAll()
console.info(current, allUsers.length)

const removeResult = await userRepo.removeById(1)
if (!removeResult.success) {
  console.error('removeById 失败:', removeResult.errorMessage)
}
```

软删除实体如果需要读取已删除记录，必须显式把 `includeDeleted` 设为 `true`，恢复后再走默认查询路径。

```ts
import { Repository } from 'ocorm'

const postRepo = new Repository('Post')

await postRepo.removeById(7)

const hidden = await postRepo.findById(7)
const deletedRow = await postRepo.findById(7, true)
console.info(hidden === null, deletedRow)

const restoreResult = await postRepo.restore(7)
if (!restoreResult.success) {
  console.error('restore 失败:', restoreResult.errorMessage)
}
```

`saveAll` 可以用于顺序保存一组实体，但它只是把每次 `save` 的结果收集起来，不保证原子性。

```ts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const userRepo = new Repository('User')

const results = await userRepo.saveAll([
  makeUser(11, 'A'),
  makeUser(12, 'B')
])

for (const [index, result] of results.entries()) {
  if (!result.success) {
    console.error(`第 ${index} 项保存失败:`, result.errorMessage)
  }
}
```

## 6. 误用示例
下面这段代码的问题不是语法，而是语义假设错误：`saveAll` 不是“自动事务 + 全成全败”。前两项可能已经提交，第三项才失败。

```ts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const userRepo = new Repository('User')

// 误用：把 saveAll 当成原子批处理
const results = await userRepo.saveAll([
  makeUser(21, 'ok-1'),
  makeUser(22, 'ok-2'),
  makeUser(22, 'duplicate-id')
])

// 错误假设：只要有一项失败，前面两项也一定回滚
console.info(results.map((item) => item.success))
```

另一个典型误用是：在事务回调里绕开 `txRepo`，继续使用外部仓储做写入。这会被 `RepositoryTransactionGuardSupport.ets` 拒绝。

```ts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const userRepo = new Repository('User')

await userRepo.transaction(async (txRepo) => {
  await txRepo.save(makeUser(31, 'inside-tx'))

  // 误用：事务里继续用外部 repo 写入
  await userRepo.save(makeUser(32, 'outside-scope-write'))
})
```

## 7. 失败语义与抛错说明
最关键的差异是：普通上下文下，很多写操作可能以“失败结果对象”的形式返回；严格事务上下文下，同样的失败会被升级为异常，从而触发回滚。

```ts
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

const userRepo = new Repository('User')

try {
  await userRepo.transaction(async (txRepo) => {
    await txRepo.save(makeUser(41, 'first'))

    // 如果这一项最终得到 success = false，Repository 会在严格事务里
    // 把它升级为 ExecutionError，进而由事务层回滚并包装成 TransactionRollbackError
    await txRepo.save(makeUser(41, 'duplicate'))
  })
} catch (e) {
  console.error('事务失败并已回滚:', e)
}
```

构造仓储时如果实体未注册，异常会在构造阶段抛出，而不是在第一次查询时才出现。

```ts
import { Repository } from 'ocorm'

try {
  const badRepo = new Repository('NotRegisteredEntity')
  await badRepo.findAll()
} catch (e) {
  console.error('实体未注册:', e)
}
```

查询侧也不是“永不抛错”。`findById` / `findAll` 过程中如果底层查询或 `afterLoad` 钩子失败，`Repository.ets` 会抛出 `ExecutionError('SELECT', ...)` 或原始钩子异常。

```ts
import { Repository } from 'ocorm'

const userRepo = new Repository('User')

try {
  await userRepo.findAll()
} catch (e) {
  console.error('查询失败:', e)
}
```

## 8. 实战规则
- 单条写入优先 `save`，并始终检查返回对象的 `success`。
- 需要排查软删除数据时，显式传 `findById(id, true)` 或其他 `includeDeleted = true` 路径。
- 需要原子性时，不要把 `saveAll` 当事务；显式包进 `transaction`，或改用 `batchInsert`。
- 事务回调里只使用回调提供的 `txRepo` 写入，不要再使用外层仓储实例。
