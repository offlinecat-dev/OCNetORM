# 05-数据库关闭类问题

> 状态：已完成  
> 适用版本：ocorm 3.x  
> 最后更新：2026-03-28

## 1. 适用范围
这篇文档只处理“数据库未初始化、已关闭后继续访问、关闭阶段日志告警、初始化连接失败”这几类问题。不要把事务超时、查询超时、关联超时混进来，那些属于其他排障专题。

```text
优先检索关键词
- 数据库未初始化，请先调用 initialize() 方法
- Database not initialized
- TestContext not initialized, call initialize() first
- 数据库 "*.db" 连接失败
- 关闭连接池失败
- 关闭数据库连接失败
- 数据库连接健康检查失败
- 数据库健康检查异常
```

## 2. 报错原文/关键词

### 2.1 抛出型错误
- `数据库未初始化，请先调用 initialize() 方法`
- `Database not initialized`
- `TestContext not initialized, call initialize() first`
- `数据库 "{dbName}" 连接失败: {reason}`

### 2.2 日志型告警
- `关闭连接池失败: {errorMessage}`
- `关闭数据库连接失败: {errorMessage}`
- `数据库连接健康检查失败`
- `数据库健康检查异常: {errorMessage}`

```ts
// 来自 errors/DatabaseError.ets 的真实错误模板
new DatabaseNotInitializedError().message
// 数据库未初始化，请先调用 initialize() 方法

new ConnectionError('ocorm.db', 'mock failure').message
// 数据库 "ocorm.db" 连接失败: mock failure
```

## 3. 真实测试场景基线
`EntryConnectionPoolSuite.ets` 没有直接做“关闭失败”用例，但它给出了数据库处于正常打开状态时的基线行为。排障时先确认你的现场有没有偏离这些基线。

- `DatabaseManager_DefaultPoolState`：默认连接池关闭，`isPoolEnabled()` 应为 `false`。
- `AcquireStoreLease_FallbackToSingleton`：连接池关闭时，`acquireStoreLease()` 应回退到单例 store。
- `WithStoreLease_AutoReleaseSingleton`：单例 lease 在回调返回后自动释放。
- `Transaction_BindsRepositoryWritesToLeasedStore`：启用连接池后，事务写入应绑定到租约 store，而不是漂移到别的连接。

```text
排障判断基线
- 如果你根本没有完成初始化，就不可能满足上述任何基线
- 如果 close() 已执行，后续再访问 getStore()/getDatabase() 本质上都会退化为“未初始化”问题
- 如果日志里出现关闭失败或健康检查异常，不要再假设当前 manager 仍然是健康可复用状态
```

## 4. 排查步骤

### 4.1 先判定是“未初始化”还是“关闭后复用”
1. 查看调用栈是否落在 `DatabaseManager.getStore()` 或 `TestContext.getDatabase()`。
2. 如果命中 `数据库未初始化，请先调用 initialize() 方法`，直接回看初始化入口是否真正执行。
3. 如果命中 `Database not initialized`，优先检查 `TestContext` 是否已经 `initialize()`，或者是否在 `close()` 之后继续复用旧实例。
4. 如果命中 `TestContext not initialized, call initialize() first`，说明连 `appContext` 都还没准备好，问题比“数据库未初始化”更早。

### 4.2 再判定是否是关闭阶段本身失败
1. 检查日志里是否有 `关闭连接池失败` 或 `关闭数据库连接失败`。
2. 这两条是 `DatabaseManager.close()` 内部捕获后记录的日志，不一定会继续向外抛异常。
3. 如果关闭日志出现后业务代码还在继续拿旧 manager 做读写，这通常是调用顺序错误，不是 ORM 自动恢复。

### 4.3 最后看连接建立和健康检查
1. 初始化时如果 `getRdbStore(...)` 或连接池建立失败，会被包装成 `数据库 "{dbName}" 连接失败: {reason}`。
2. 运行期如果健康检查失败，会看到 `数据库连接健康检查失败` 或 `数据库健康检查异常: ...`。
3. 健康检查异常说明 manager 已把内部健康状态切成不健康，后续应优先恢复连接，而不是继续压测当前实例。

## 5. 复现片段

### 5.1 初始化前直接访问 TestContext
```ts
import { TestContext } from '../testcore/TestContext'

const ctx = new TestContext()
ctx.getAppContext()
// Error: TestContext not initialized, call initialize() first

ctx.getDatabase()
// Error: Database not initialized
```

### 5.2 关闭后继续访问 DatabaseManager
```ts
import { DatabaseManager } from 'ocorm'

const manager = DatabaseManager.getInstance()
await manager.close()
manager.getStore()
// DatabaseNotInitializedError
// 数据库未初始化，请先调用 initialize() 方法
```

### 5.3 TestContext 关闭后复用旧实例
```ts
const ctx = await TestContext.create(context, {
  databaseName: 'close-case.db'
})

await ctx.close()
await ctx.executeSql('SELECT 1')
// Error: Database not initialized
```

## 6. 修复建议

### 6.1 初始化入口只保留一个
- 应用侧把 `OCORMInit(...)` 或等价初始化放到确定只执行一次的启动入口。
- 测试侧统一走 `TestContext.create(...)` 或 `await ctx.initialize(...)`，不要一部分 case 手写，一部分 case 偷用旧上下文。

### 6.2 close() 之后禁止复用旧对象
- `DatabaseManager.close()` 会把 `store` 置空、`initialized` 置为 `false`、`healthy` 置为 `false`。
- `TestContext.close()` 也会把本地 `db` 置空并重置初始化状态。
- 所以 close 后继续拿旧引用操作数据库，不是“偶发失败”，而是确定性的生命周期违规。

### 6.3 把关闭日志当成失败信号，不要当普通噪声
- `关闭连接池失败`、`关闭数据库连接失败` 说明资源回收链路已经不干净。
- `数据库连接健康检查失败`、`数据库健康检查异常` 说明当前连接状态不可靠。
- 这类日志出现后，优先重建 manager / context，再排查底层 store 或环境问题。

### 6.4 用测试基线回看现场
- 如果你的现场不满足 `EntryConnectionPoolSuite` 里的正常基线，就先修生命周期，不要先怀疑业务 SQL。
- 尤其是 `AcquireStoreLease_FallbackToSingleton` 和 `WithStoreLease_AutoReleaseSingleton` 这两个场景，可以快速判断 manager 是否还处在一个可用的打开态。

```text
修复优先级
1. 先保证 initialize() 真的执行且只执行一次
2. 再保证 close() 后不复用旧 TestContext / DatabaseManager
3. 再处理连接建立失败的环境原因
4. 最后处理关闭失败或健康检查异常的底层 store 问题
```
