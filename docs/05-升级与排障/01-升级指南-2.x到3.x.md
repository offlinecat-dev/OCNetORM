# 01-升级指南-2.x到3.x

> 状态：已完成
> 适用版本：从 2.x 升级到当前仓库 3.0.2
> 最后更新：2026-03-28

## 1. 文档边界
这份指南只基于仓库内可验证材料编写：
- `OCORM/CHANGELOG.md`
- `OCORM/README.md`
- `OCORM/docs/20-源码开发指南/00-导航与约定` 到 `14-测试与验收` 的现有文档

结论分两类：
- 可以直接由当前源码证明的 3.x 行为
- 无法从仓库保留代码完整回放 2.x 时，明确标注为“从当前仓库可推断的升级点”

## 2. 升级结论
3.0 不是“重写调用方式”，而是“保留主体 API，收紧默认行为和安全语义”。

从当前仓库可推断，2.x 到 3.x 升级时最容易出问题的不是 `Repository`、`QueryBuilder`、`EntityData` 这些入口是否存在，而是以下默认行为是否仍被旧代码依赖：
- `DatabaseConfig.encrypt` 默认值变化
- 事务隔离级别与只读事务从“可尝试/可降级”收紧为更明确的 fail-fast 语义
- 聚合 DSL 与实体查询模式被彻底分离
- 原生 SQL 在当前 3.0.2 上默认强制参数化
- 升到当前 3.0.2 时，测试代码若碰过 `DatabaseManager` 内部测试注入 API，也要一起迁移

## 3. 升级前先做一次仓库盘点
建议先扫一遍高风险调用点，再开始改业务代码。

```bash
rg -n "rawQuery\(|rawQuerySafe\(|rawExecute\(|selectRaw\(|groupBy\(|having\(|serializable\(|REPEATABLE_READ|SERIALIZABLE|readOnly|new DatabaseConfig\(" OCORM entry
```

重点看四类代码：
- 依赖数据库默认加密配置的初始化代码
- 把 `selectRaw/groupBy/having` 和 `get/getPaginated/count` 混用的查询代码
- 依赖 `SERIALIZABLE` 或 `REPEATABLE_READ` 的事务代码
- 非参数化原生 SQL 与旧测试钩子用法

## 4. 从当前仓库可推断的升级点
### 4.1 默认加密值已变化
- `encrypt: boolean = true`

这意味着如果你的 2.x 代码没有显式写出 `encrypt`，升级后不要再假设默认值和旧版本一致。

### 4.2 事务能力不是“枚举里有就能用”
当前源码 `TransactionOptions.ets` 仍定义了：
- `READ_UNCOMMITTED`
- `READ_COMMITTED`
- `REPEATABLE_READ`
- `SERIALIZABLE`

但当前真实行为由 `TransactionManager.ets` 决定：
- `READ_COMMITTED` 支持
- `READ_UNCOMMITTED` 条件支持
- `REPEATABLE_READ` / `SERIALIZABLE` 在执行期显式拒绝
- `TransactionOptions.serializable()` 在创建时直接抛错
- `readOnly` 依赖 `PRAGMA query_only`，平台不支持时 fail-fast

### 4.3 聚合 DSL 与实体模式已经分家
当前 3.x 的 `QueryExecutor.ets` 会在实体模式直接抛错：
- `实体查询不支持 selectRaw/groupBy/having，请使用 getRaw() 或 aggregate()`

因此，旧代码如果把 `selectRaw/groupBy/having` 和 `get/getPaginated/count` 混在一起，升级后要改写。

### 4.4 原生 SQL 到当前 3.0.2 时必须参数化
当前仓库 `Repository.ets` 与 `rawsql/RawSqlGuards.ets` 已明确：
- `rawQuery`
- `rawQuerySafe`
- `rawExecute`

都会校验 `?` 占位符数量，并拒绝参数数目不匹配的 SQL。`CHANGELOG.md` 的 `3.0.2` 也把 `rawQuery` 默认强制参数化列为显式行为变化。

## 5. 升级前后代码对照
### 5.1 数据库初始化：显式声明加密策略
下面的“升级前”写法不是对 2.x 历史源码的逐行还原，而是从 `CHANGELOG` 可推断出的常见风险写法：依赖默认值。

```ts
// 升级前：从当前仓库可推断的风险写法
// 风险点：如果 2.x 代码依赖旧默认值，升级到 3.x 后默认 encrypt 已变为 true
const config = new DatabaseConfig('app.db')
await OCORMInit(context, { config })
```

```ts
// 升级后：把意图写死，不再依赖默认值
// 需要保持旧行为时，显式传 false；接受 3.x 默认安全策略时，则显式传 true 或省略但写入说明
const config = new DatabaseConfig('app.db', undefined, false)
await OCORMInit(context, { config })
```

### 5.2 查询模式：聚合查询改用 `getRaw()` / `aggregate()`
这部分是当前 3.x 源码可直接证明的升级要求。

```ts
// 升级前：旧代码中常见的混用模式
const rows = await new QueryExecutor(
  repo.createQueryBuilder()
    .selectRaw(['status', 'COUNT(*) AS total'])
    .groupBy('status')
    .having('COUNT(*) > ?', [1])
).get()
```

```ts
// 升级后：切到原生/聚合模式
const rows = await new QueryExecutor(
  repo.createQueryBuilder()
    .selectRaw(['status', 'COUNT(*) AS total'])
    .groupBy('status')
    .having('COUNT(*) > ?', [1])
).getRaw()
```

### 5.3 事务配置：删除对 `SERIALIZABLE` 的幻想
当前 `TransactionOptions.serializable()` 直接抛错，这不是建议，而是运行时事实。

```ts
// 升级前：旧代码若这样写，升级到当前 3.x 会在创建选项时直接失败
const options = TransactionOptions.serializable()
await repo.transactionWithOptions(async (txRepo) => {
  await txRepo.save(user)
}, options)
```

```ts
// 升级后：改成当前仓库真正支持的隔离级别
const options = TransactionOptions.fromConfig({
  isolation: IsolationLevel.READ_COMMITTED,
  timeout: 10000,
  retries: 1,
  retryDelay: 200,
  readOnly: false
})

await repo.transactionWithOptions(async (txRepo) => {
  await txRepo.save(user)
}, options)
```

### 5.4 升到当前 3.0.2 时：测试钩子从 `DatabaseManager` 主类迁出

```ts
manager.applyRuntimeConfigForTesting(config)
```

## 6. 误用示例
下面这些写法升级到当前 3.0.2 后都会出问题。

```ts
// 误用 1：非参数化 rawQuery
await repo.rawQuery("SELECT * FROM user WHERE name = 'Alice'")

// 误用 2：实体模式混用聚合 DSL
await new QueryExecutor(
  repo.createQueryBuilder()
    .selectRaw(['COUNT(*) AS total'])
).get()

// 误用 3：继续使用当前实现不支持的隔离级别
await repo.transactionWithOptions(async (txRepo) => {
  await txRepo.findAll()
}, TransactionOptions.fromConfig({ isolation: IsolationLevel.REPEATABLE_READ }))
```

对应后果：
- 误用 1：会被 `RAW_QUERY` 参数化守卫拒绝
- 误用 2：会抛实体模式与聚合 DSL 混用错误
- 误用 3：会抛事务隔离级别不支持错误

## 7. 推荐升级顺序
1. 先把 `DatabaseConfig` 显式化，尤其是 `encrypt`
2. 再改事务配置，移除 `SERIALIZABLE` / `REPEATABLE_READ` 依赖
3. 再清理聚合查询，把 `get()` 改为 `getRaw()` / `aggregate()`
4. 最后收口所有原生 SQL，统一改成参数化

这个顺序的原因很现实：
- 初始化与事务问题会先阻断应用启动和核心写入路径
- 查询模式问题通常集中在少数报表/统计接口
- 原生 SQL 和测试钩子更适合最后一次性清理

## 8. 兼容性检查清单
升级完成后，至少核对下面这些项。

- [ ] 所有 `new DatabaseConfig(...)` 都已明确评估 `encrypt` 行为
- [ ] 不再使用 `TransactionOptions.serializable()`
- [ ] 不再依赖 `REPEATABLE_READ` / `SERIALIZABLE`
- [ ] `readOnly` 事务调用方已准备好处理 fail-fast
- [ ] 所有 `rawQuery/rawQuerySafe/rawExecute` 都是参数化 SQL
- [ ] 所有 `selectRaw/groupBy/having` 查询都改到 `getRaw()` 或 `aggregate()`

## 9. 验收建议
最少做三类回归：
- 初始化回归：确认数据库能按预期打开，且 `encrypt` 配置符合业务要求
- 事务回归：确认 `READ_COMMITTED` / `READ_UNCOMMITTED` 的业务路径能跑通，错误隔离级别会按预期 fail-fast
- 查询与安全回归：确认统计类查询改到 `getRaw()` 后结果不变，所有原生 SQL 都已参数化

```bash
ohpm install
hvigor test
```

如果你有自定义集成套件，建议额外补一组最小人工验证：
- 打开应用
- 跑一次写入事务
- 跑一次聚合查询
- 跑一次原生 SQL 查询
