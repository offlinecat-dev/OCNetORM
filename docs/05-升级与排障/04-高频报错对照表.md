## 1. 使用说明
这份对照表不是概念文档，只用于日志检索和首轮排障。先按“报错原文/关键词”命中日志，再按“复现片段”确认触发条件，最后按“修复建议”收敛代码改动。

```text
建议优先检索的关键词
- 数据库未初始化
- Database not initialized
- 事务超时
- 后台清理
- 同一 QueryExecutor 实例不支持并发复用
- 查询超时
- 查询条件无效
- 非法排序方向
- 关联关系
```

## 2. 高频报错速查

| 场景 | 报错原文/关键词 | 首查文件 | 第一判断 |
| --- | --- | --- | --- |
| 数据库未完成初始化或已被关闭 | `数据库未初始化，请先调用 initialize() 方法` / `Database not initialized` | `errors/DatabaseError.ets`、`DatabaseManager.ets` | 先看 `initialize()` 是否执行过，再看 `close()` 是否提前跑过 |
| 查询执行器被并发复用 | `同一 QueryExecutor 实例不支持并发复用，请为并发操作创建独立实例` | `QueryExecutor.ets` | 同一个 `QueryExecutor` 是否被 `Promise.all` 或重入复用 |
| 查询或关联加载超时 | `查询超时 (Nms)` / `${operation} 查询超时 (Nms)` | `QueryExecutor.ets`、`RelationLoader.ets`、`QueryExecutorSqlSupport.ets` | 先区分主查询超时还是关联查询超时 |
| 查询条件或排序参数非法 | `查询条件无效: ...` / `非法排序方向: ...` | `errors/QueryError.ets`、`QueryExecutorSqlSupport.ets` | 多数是 `having`、`orderBy`、动态拼条件传入了不受支持值 |
| 关联未注册 | `实体 'User' 的关联关系 'posts' 未找到` / `关联 'posts' 未注册` | `errors/RelationError.ets`、`RelationLoader.ets`、`QueryExecutorRelationSupport.ets` | 先看元数据注册是否完整，再看 relation path 是否拼错 |

## 3. 数据库初始化/关闭类

### 报错原文/关键词
- `数据库未初始化，请先调用 initialize() 方法`
- `Database not initialized`

### 排查步骤
1. 检查是否已经执行 `OCORMInit(...)`。
2. 检查当前调用点之前是否执行过 `DatabaseManager.getInstance().close()`。
3. 如果日志里只有英文错误，优先核对底层调用链是否直接命中 `DatabaseManager.getStore()`。

### 复现片段
```ts
import { DatabaseManager } from 'ocorm'

DatabaseManager.getInstance().getStore()
// DatabaseNotInitializedError: 数据库未初始化，请先调用 initialize() 方法
```

### 修复建议
- 应用侧统一从一个初始化入口调用 `OCORMInit(...)`，不要把首次初始化分散到页面生命周期里。
- 如有自定义测试封装，先完成初始化再调用任何数据库读写。
- `close()` 之后不要复用旧的 `DatabaseManager` 状态，重新初始化一遍。

## 4. 事务超时、回滚与重叠调用

### 报错原文/关键词
- `事务已回滚: 事务超时 (50ms)`
- `事务超时，转入后台清理: 50ms`
- `事务超时后后台清理已完成`
- 关键词：`嵌套调用`、`跨 Repository`

### 排查步骤
1. 先确认事务是否使用了 `TransactionOptions.withTimeout(...)` 或自定义 `timeout`。
2. 看首个事务回调里是否有未释放的 Promise、未返回的异步任务或长时间等待信号量。
3. 如果第二个事务立即失败，不要先怀疑锁泄漏，先检查 `TransactionManager.hasAnyActiveTransaction()` 对应的后台清理是否还没结束。
4. 如果配置了重试，核对是否是“超时”而不是“可重试失败”；超时场景默认不应重试。

### 复现片段
```ts
const repo = new Repository('User')
const gate = new Promise<void>(() => {})

await repo.transactionWithOptions(async () => {
  await gate
}, TransactionOptions.withTimeout(50))

// 真实实现会抛 TransactionRollbackError，核心关键词为：
// 事务超时 (50ms)
```

### 修复建议
- 缩短事务内部逻辑，只保留真正需要原子性的写操作。
- 不要在事务内等待外部 UI 信号、网络回包或长时间定时器。
- 超时后不要立刻复用同一个 Repository 再开事务，先等待后台清理完成。
- 对连接池/多仓库写入场景，复用事务回调里传入的 `txRepo`，不要在事务中再 new 一个未绑定上下文的 `Repository`。

## 5. QueryExecutor 并发复用与查询超时

### 报错原文/关键词
- `同一 QueryExecutor 实例不支持并发复用，请为并发操作创建独立实例`
- `查询超时 (20ms)`
- `${operation} 查询超时 (20ms)`

### 排查步骤
1. 检查同一个 `QueryExecutor` 实例是否被多个异步分支同时调用 `get()` / `count()` / `getRaw()`。
2. 检查是否设置了查询级超时 `builder.timeout(...)`，或数据库级超时 `DatabaseConfig.setQueryTimeout(...)`。
3. 如果日志提示超时，但结果集随后又回来了，优先检查 `QueryExecutor` 的“迟到结果集关闭”逻辑。
4. 如果是关联加载场景，还要继续看 `RelationLoader.ets` 的超时逻辑，而不是只盯主查询。

### 复现片段
```ts
const builder = new QueryBuilder('User').timeout(20)
const executor = new QueryExecutor(builder)

QueryExecutor.setResultSetDelayMsForTest(80)
await executor.get()
// 查询超时 (20ms)

await Promise.all([executor.get(), executor.count()])
// 同一 QueryExecutor 实例不支持并发复用，请为并发操作创建独立实例
```

### 修复建议
- 每个并发查询分支创建独立的 `QueryExecutor` 实例，不要共享同一个执行器对象。
- 把超时值与真实数据量对齐；20ms 这类值只适合测试钩子，不适合常规业务查询。
- 超时后优先排查 ResultSet 生成慢、关系展开慢、子查询分批多，而不是盲目放大线程或重试。
- 对慢查询场景优先裁剪 `with`、`withCount`、`whereExists`、分页与排序组合。

## 6. 查询条件非法与关联未注册

### 报错原文/关键词
- `查询条件无效: ...`
- `非法排序方向: DESC NULLS FIRST`
- `实体 'User' 的关联关系 'posts' 未找到`
- `关联 'posts' 未注册`

### 排查步骤
1. 先核对动态拼装的 `orderBy` / `having` / relation path 是否来自白名单。
2. 如果是 `having` 或 `orderBy` 输入异常，优先比对 `QueryExecutorSqlSupport.ets` 的参数校验逻辑。
3. 如果是关联未注册，先确认实体、列、关系三者是否都完成元数据注册，再检查调用端 relationName 是否拼写一致。
4. 如果问题只出现在嵌套关联路径，继续检查 `QueryExecutorRelationSupport.ets` 的 path 拼接结果。

### 复现片段
```ts
new QueryBuilder('User').orderBy('name', 'DESC NULLS FIRST' as 'ASC')
// InvalidConditionError
// 查询条件无效: 非法排序方向: DESC NULLS FIRST

const loader = new RelationLoader('User')
await loader.loadRelation([userEntity], 'posts')
// RelationNotFoundError
// 实体 'User' 的关联关系 'posts' 未找到
```

### 修复建议
- 所有排序方向只允许 `ASC` / `DESC` 两个固定值，不接受透传 SQL 片段。
- `having`、`whereExists`、relation path 一律走白名单或枚举，不要信任外部字符串输入。
- 关联查询前先完成 `MetadataStorage.registerEntity`、`registerColumn`、`registerRelation` 的成组注册。
- 对嵌套路径如 `posts.comments.author`，按段检查每一级 relation 是否存在，不要一次性猜整体链路。

## 7. 最小自检清单

```text
- 初始化前是否直接调用了 DatabaseManager.getStore()
- 事务超时后是否等待后台清理完成再发起下一次事务
- 是否把同一个 QueryExecutor 实例拿去并发复用
- 查询超时是主查询超时，还是 RelationLoader 的关联查询超时
- orderBy / having / relationName 是否来自受控白名单
```
