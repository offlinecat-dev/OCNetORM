# 01-验证系统总览

> 状态：完成
> 适用版本：ocorm 3.x（基于当前源码）
> 最后更新：2026-03-27

## 1. 核心目标
验证系统用于在实体写入前统一执行字段规则，并以 `ValidationError` 家族错误提供可定位的失败信息。
钩子系统用于在实体生命周期节点执行同步/异步逻辑，并在失败时抛出 `HookExecutionError`。
两者都是元数据驱动：规则来自 `验证规则存储`，钩子来自实体元数据中的 `EntityHooks`。

## 2. 关键组件与职责
- `validation/ValidationDecorators`：声明式注册规则（`Required`、`Length`、`Email` 等）
- `验证规则存储`：按 `entityName -> propertyName -> ValidationRule[]` 存储规则
- `validation/EntityValidator`：执行规则并返回 `ValidationResult`，或通过 `validateOrThrow` 抛错
- `EntityHooks`：定义 `HookName`、`HookCallback`、`GlobalHookCallback` 与 `EntityHooks`
- `HooksProcessor`：执行实体钩子和全局钩子，处理 before/after 执行顺序
- `ValidationError`：字段级与实体级验证错误
- `HookError`：钩子执行错误

## 3. 规则正例（验证 + 钩子）
```ts
import { ColumnType, Email, Length, Range, Required, defineEntity } from 'ocorm'

defineEntity('User', {
  columns: [
    { property: 'name', type: ColumnType.TEXT },
    { property: 'email', type: ColumnType.TEXT },
    { property: 'age', type: ColumnType.INTEGER }
  ]
})

class User {
  @Required()
  @Length({ min: 2, max: 30 })
  name: string = ''

  @Required()
  @Email()
  email: string = ''

  @Range(18, 65)
  age: number = 18
}
```

```ts
import { EntityValidator } from 'ocorm'

const result = EntityValidator.validate('User', entityData)
if (!result.isValid()) {
  // 可批量查看
  const summary = result.getSummary()
}

// 需要失败即中断时
EntityValidator.validateOrThrow('User', entityData)
```

```ts
import { HooksProcessor } from 'ocorm'

const hooks = HooksProcessor.getInstance()
await hooks.executeBeforeSave('User', entityData)
// 持久化动作...
await hooks.executeAfterSave('User', entityData)
```

## 4. 误用示例
### 4.1 在未注册实体上声明验证装饰器
`ValidationDecorators.ensureEntityRegistered` 会检查 `实体元数据注册表.hasEntity(entityName)`，未注册会直接抛 `EntityNotRegisteredError`。也就是说，当前实现里如果你要用 `@Required()` 这类装饰器，实体元数据必须先通过 `defineEntity(...)`、`registerEntity(...)` 等方式注册。

```ts
class NotRegisteredUser {
  @Required()
  name: string = ''
}
// 如果实体元数据未注册，装饰阶段即失败
```

### 4.2 误以为 `Unique` 自动查库
`unique` 规则只有在 `ValidationOptions.uniqueChecker` 存在时才执行校验。

```ts
// 误用：没有提供 uniqueChecker，Unique 不会触发实际唯一性判断
EntityValidator.validate('User', entityData)

// 正确：注入 uniqueChecker
EntityValidator.validate('User', entityData, {
  uniqueChecker: (entityName, propertyName, value) => {
    return value !== 'duplicated@example.com'
  }
})
```

## 5. 报错与定位方式
### 5.1 验证报错定位
- 单一错误：`validateOrThrow` 直接抛具体子类（如 `RequiredValidationError`）
- 多错误：抛 `EntityValidationError`，`reason` 为多条 message 拼接
- 定位字段：`ValidationError` 上下文包含 `entityName`、`columnName`、`operation='数据校验'`

```ts
try {
  EntityValidator.validateOrThrow('User', entityData)
} catch (error) {
  // 常见类型：RequiredValidationError / EmailValidationError / EntityValidationError
  // 推荐记录 error.name + error.message
}
```

### 5.2 钩子报错定位
- 任何 hook 回调抛错会包装为 `HookExecutionError`
- 上下文包含 `entityName`、`operation=hookName`、`details=已清洗错误信息`
- `before*` 顺序：全局钩子 -> 实体钩子；`after*` 顺序：实体钩子 -> 全局钩子

```ts
try {
  await HooksProcessor.getInstance().executeBeforeInsert('User', entityData)
} catch (error) {
  // HookExecutionError: 可根据 operation 判断是 beforeInsert/afterUpdate 等哪个阶段失败
}
```

## 6. 最小排障清单
1. 先确认实体是否已注册到 `实体元数据注册表`。
2. 再检查验证规则是否注册到了 `验证规则存储`（实体名/属性名是否一致）。
3. `Unique` 失败或未生效时，优先检查 `uniqueChecker` 是否传入。
4. 钩子异常时，先看 `HookExecutionError` 的 `hookName`（operation）再定位具体回调。
