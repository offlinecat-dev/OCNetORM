# 事务守卫：嵌套、跨仓库、未绑定写入

> 源码对齐：`RepositoryTransactionGuardSupport.ets`、`Repository.ets`、`TransactionManager.ets`

## 1. 契约总览

### `ensureWritePathAllowed(operation)`（写入入口守卫）

| 项 | 契约 |
|---|---|
| 触发入口 | `save/remove/forceRemove/restore/saveAll/batchInsert/attach/detach/sync/rawExecute` |
| 放行条件 | 当前仓储处在严格事务上下文（`strictDepth > 0`） |
| 拒绝条件 A | 连接池模式 + 存在活动事务 + 当前仓储未绑定事务上下文 |
| 拒绝条件 B | 当前连接已有活动事务，但调用发生在事务边界外 |
| 异常 | `TransactionRollbackError` |

### `ensureNoNestedTransaction(entryName)`（启动事务守卫）

| 项 | 契约 |
|---|---|
| 触发入口 | `transaction(...)`、`transactionWithOptions(...)` |
| 拒绝条件 A | 同一 `Repository` 在事务回调内再次启动事务 |
| 拒绝条件 B | 连接池模式下存在活动事务，却用未绑定仓储再次启动事务 |
| 拒绝条件 C | 另一个仓储在外层事务执行期间尝试启动事务（跨仓库嵌套） |
| 异常 | `TransactionRollbackError` |

## 2. 上下文限制（必须理解）
- 只要进入事务回调，写入必须使用回调参数 `txRepo`。
- `setPooledWriteGuardMode('global')` 时，连接池下会启用全局未绑定写入保护。
- 当前实现不提供仓储级自动 `SAVEPOINT` 语义，嵌套事务会被显式拒绝。

## 3. 可运行示例（正确事务上下文）

```arkts
import { Context } from '@kit.AbilityKit'
import {
  ColumnType,
  DatabaseConfig,
  EntityData,
  EntityDataInput,
  OCORMInit,
  Repository,
  defineEntity
} from 'ocorm'

async function bootstrap(context: Context): Promise<void> {
  defineEntity('User', {
    tableName: 'users',
    columns: [
      { property: 'id', name: 'id', type: ColumnType.INTEGER, primaryKey: true, autoIncrement: false },
      { property: 'name', name: 'user_name', type: ColumnType.TEXT }
    ]
  })

  await OCORMInit(context, {
    config: new DatabaseConfig('tx_guard.db').setPooledWriteGuardMode('global'),
    autoCreateTables: true
  })
}

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

export async function runTxGuardHappyPath(context: Context): Promise<void> {
  await bootstrap(context)
  const repo = new Repository('User')

  await repo.transaction(async (txRepo) => {
    await txRepo.save(makeUser(1, 'inside-tx'))
    await txRepo.rawExecute('UPDATE users SET user_name = ? WHERE id = ?', ['updated', 1])
  })
}
```

## 4. 失败案例 A：同仓库嵌套事务

```arkts
import { Context } from '@kit.AbilityKit'
import { Repository } from 'ocorm'

export async function failNestedTx(context: Context): Promise<void> {
  await bootstrap(context)
  const repo = new Repository('User')
  try {
    await repo.transaction(async (txRepo) => {
      await txRepo.transaction(async () => {
      })
    })
  } catch (e) {
    // TransactionRollbackError: 不支持同一 Repository 嵌套调用 transaction
    console.error('expected nested reject:', e)
  }
}
```

## 5. 失败案例 B：未绑定事务上下文写入

```arkts
import { Context } from '@kit.AbilityKit'
import { EntityData, EntityDataInput, Repository } from 'ocorm'

function makeUser(id: number, name: string): EntityData {
  const input = EntityDataInput.create()
  input.set('id', id)
  input.set('name', name)
  return EntityData.from('User', input)
}

export async function failUnboundWrite(context: Context): Promise<void> {
  await bootstrap(context)
  const outerRepo = new Repository('User')
  try {
    await outerRepo.transaction(async (txRepo) => {
      await txRepo.save(makeUser(10, 'ok'))
      await outerRepo.save(makeUser(11, 'blocked')) // 故意误用
    })
  } catch (e) {
    // TransactionRollbackError: 检测到...未绑定事务上下文...禁止写入
    console.error('expected unbound write reject:', e)
  }
}
```

## 6. 排障顺序
1. 先确认调用对象是否为回调入参 `txRepo`。
2. 再确认是否在另一个仓储上开启了新的 `transaction/transactionWithOptions`。
3. 最后检查连接池与 `pooledWriteGuardMode` 设置是否触发全局保护。
